## Gap 1: [high] web_fetch tool does not use LLM for prompt-guided answering

- **Category**: incomplete
- **Section**: §四 工具系统 — 内置工具清单 (web_fetch)
- **Files**: myagent/tools/builtin/web_tools.py (lines 28–65, 70–164)
- **Description**: The design spec explicitly states that `web_fetch` should "fetch a URL, convert the page to markdown, and **answer `prompt` against it using a small fast model**". The current implementation uses a regex-based keyword extraction function `_extract_relevant()` that scores lines by keyword overlap with the prompt — it does not call any LLM (small or otherwise) to actually answer the prompt against the fetched content.
- **Expected**: After fetching and converting HTML to markdown/text, the tool should pass the content and the user's prompt to a lightweight LLM call (e.g., Non-think mode on the configured provider, or a dedicated small model) and return the model's answer. This is a key differentiator from a plain "fetch and grep" tool.
- **Actual**: `_extract_relevant()` selects up to 80 lines with the highest keyword-overlap score against the prompt. No LLM is invoked. The output is a plain concatenation of extracted lines rather than a reasoned answer.

## Gap 2: [medium] config_set `value` parameter schema missing `type` field

- **Category**: incomplete
- **Section**: §四 工具系统 — 统一接口 (JSON Schema completeness)
- **Files**: myagent/tools/builtin/config_tools.py (lines 131–153)
- **Description**: The JSON Schema for `config_set`'s `value` parameter defines only a `description` text block with no `type` field. All other built-in tools define proper `type` fields for every parameter (e.g., `"type": "string"`, `"type": "boolean"`). The spec's unified Tool protocol requires `parameters: dict  # JSON Schema (OpenAI function calling 格式)`. A missing `type` makes the schema underspecified for the model and for any downstream consumers of the tool definition.
- **Expected**: The `value` parameter should carry a `type` field. Since it accepts heterogeneous values (string, int, float, bool, list), the appropriate JSON Schema approach would be `"type": ["string", "number", "boolean", "array", "object"]` or a similar multi-type declaration, so the model understands the parameter's range.
- **Actual**: `{"description": "...", }` — no `type` key present. The code-side validation (`_validate_value`) handles coercion at runtime, but the schema presented to the LLM is incomplete.

## Gap 3: [medium] mcp_read_resource, mcp_get_prompt, and config_set are not assigned explicit permission levels

- **Category**: deviation
- **Section**: §五 权限/沙箱系统 — 权限分级 (Level 0–3 mapping)
- **Files**: myagent/permissions/controller.py (lines 32–46, TOOL_LEVEL_MAP)
- **Description**: The `TOOL_LEVEL_MAP` dictionary assigns every built-in tool from the spec an explicit level (0–2), plus `send_message` at level 0. However, three tools registered at startup are absent from this map:
  - `mcp_read_resource` — reads MCP resource data (read-only, should be level 0)
  - `mcp_get_prompt` — invokes an MCP prompt template (read-only, should be level 0)
  - `config_set` — adjusts runtime configuration (in-memory write, should be level 1)

  Because `_get_level()` returns `TOOL_LEVEL_MAP.get(tool_name, 3)`, all three fall through to the default of level 3 (network_write). This means the user sees a permission prompt for these tools with a misleading "network_write" level label, and auto_allow levels=[] does not cover them.
- **Expected**: `mcp_read_resource` and `mcp_get_prompt` should map to level 0 (read). `config_set` should map to level 1 (write — it modifies agent behavior at runtime, analogous to `memory_write`). All three should be added to `TOOL_LEVEL_MAP`.
- **Actual**: The three tool names are missing from `TOOL_LEVEL_MAP`. They silently inherit level 3 via the `dict.get(..., 3)` fallback.

## Gap 4: [medium] Dream engine factual-error detection does not correct memories — it only flags them

- **Category**: incomplete
- **Section**: §六 记忆系统 + 梦境机制 — 梦境机制 (纠正错误记忆)
- **Files**: myagent/memory/dream.py (lines 934–1076, `_check_factual_errors`)
- **Description**: The design spec states that the dream cycle should identify and **correct** erroneous memories ("纠正错误记忆" in the flowchart at §六). The `_check_factual_errors()` method (gap-15-04 implementation) detects factual discrepancies between individual memory file content and the project environment detected at startup (e.g., a memory claiming "Python 3.8" when the project uses "Python 3.12", or referencing `flake8` when the project uses `ruff`). However, it only appends these discrepancies as human-readable strings to the dream log's analysis section. It does **not** update the memory file, does not mark it as stale, does not create a correction note, and does not feed the discrepancy back into the sub-agent prompt for LLM-driven correction.
- **Expected**: When a factual discrepancy is detected in a single memory file (not just a pair-wise contradiction), the dream engine should either: (a) update the memory file content to correct the factual error, or (b) in the sub-agent path, include the detected discrepancies in the sub-agent prompt so the LLM can apply corrections via `memory_write` calls. The current "flag only" behavior means the user is never automatically informed and the erroneous memory persists.
- **Actual**: `_check_factual_errors()` returns a list of strings that are appended to `analysis_sections` in `_run_inline()`. No memory write or update operation is triggered. In the sub-agent path (`_run_as_subagent`), the detected factual errors are never passed into the sub-agent prompt — the sub-agent only receives the raw memory summary and transcript excerpts, without the inline-detected discrepancies.

## Gap 5: [low] L0 system prompt hardcodes model name "DeepSeek V4 Pro"

- **Category**: deviation
- **Section**: §三 上下文分层结构 — L0 System Prompt (固定，不可驱逐)
- **Files**: myagent/context/builder.py (line 64, `L0_SYSTEM_PROMPT`)
- **Description**: The L0 system prompt begins with "You are MyAgent, a CLI-based AI assistant powered by DeepSeek V4 Pro." This string is a class-level constant and is never updated when a different model is configured. The spec's ModelConfig allows `provider` and `model` to be set to any LiteLLM-supported model (including fallback models), and the config system supports `model.provider` and `model.model` overrides. If a user configures `model: gpt-4o` (via config.yaml or runtime override), the system prompt will still claim the agent is "powered by DeepSeek V4 Pro", which is factually misleading and could confuse the model about its own capabilities (e.g., context window size expectations).
- **Expected**: The L0 system prompt should either use a generic phrasing ("powered by a large language model") or be dynamically templated with the configured model name. If dynamic, it should be constructed at context-build time from `self.config.model.model` or `self.config.model.provider`.
- **Actual**: The class constant `L0_SYSTEM_PROMPT` is a static string never interpolated with actual model configuration.
