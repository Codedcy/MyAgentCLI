# Gap Report — Round 12

## Summary

After exhaustively reading all 50+ Python files under `myagent/` and comparing them against all 11 sections of the design spec, the codebase is remarkably well-aligned with the specification. The four-layer architecture, full ReAct loop, six-layer context model, 4-layer progressive compression, 14+ built-in tools, 4-level permission system, file-level memory system with dream engine, 3-tier skill system, sub-agent pool with lifecycle management, 7-level config merge, project-aware session system, and async-safe logging are all substantially implemented.

The 8 gaps below are minor deviations, incomplete guardrails, and edge-case oversights. No critical functionality is missing.

- 缺失功能 (missing): 0
- 不完整实现 (incomplete): 5
- 偏离设计 (deviation): 3

## Gap 1: hardware-based 1M context window in usage estimation
- **Category**: deviation
- **Section**: §三 上下文分层
- **Files**: `myagent/agent/engine.py`
- **Description**: The `_estimate_context_usage` method hardcodes `context_window = 1_000_000` (DeepSeek V4 Pro nominal size) instead of deriving it from the active model configuration. The spec states "上下文窗口 (跟随模型上限)" — the window size follows the model's upper limit. If a different model with a smaller window is configured (or a fallback model is active), the usage-percentage estimate will be inaccurate, potentially causing late compression triggers.
- **Expected**: The context window size should be read from the model configuration or a model capability table (e.g., `context_window = config.model.context_window` or a lookup based on `config.model.model`), so that compression thresholds are computed correctly regardless of which model is in use.
- **Actual**: `context_window = 1_000_000` is hardcoded on line 935 of engine.py, with a comment calling it "DeepSeek V4 Pro nominal context".

## Gap 2: minimum_savings debounce applied only to Layer 2, not overall compression
- **Category**: deviation
- **Section**: §三 压缩策略（四层渐进）— 防抖保护
- **Files**: `myagent/context/compression.py`
- **Description**: The spec states "压缩后 token 减少 < 10% → 跳过", which reads as a guard on the overall compression result — if all layers together produce less than 10% savings, the entire compaction should be skipped. The code applies this `minimum_savings` check exclusively within Layer 2 (`_layer2_summarize`), so Layer 1 and Layer 3 always apply their changes regardless of total effectiveness. Layer 1 is zero-cost so this is benign, but Layer 3 could produce negligible savings yet still consumes an LLM API call.
- **Expected**: The 10% savings check should be computed on the total size delta of all applied layers, and if total savings fall below `minimum_savings`, the compacted result should be discarded in favor of the original messages (or the least-effective layers should be rolled back).
- **Actual**: Only Layer 2 checks `savings >= min_savings` (line 77 of compression.py). Layers 1 and 3 always apply when their conditions are met.

## Gap 3: agent summarizer input truncated at 20000 characters
- **Category**: incomplete
- **Section**: §四 工具系统 — 工具调用流程
- **Files**: `myagent/agent/engine.py`
- **Description**: The spec states large tool results (>5000 chars) are sent to a sub-agent for summarization, producing a "summary + file reference". The implementation in `_summarize_via_subagent` truncates the input text to 20000 characters (`result.output[:20000]`) before passing it to the summarization sub-agent. For extremely large results (100K+ chars originating from large file reads, MCP resource fetches, or web page dumps), the summarizer only sees the first 20% (or less) of the content and cannot summarize the remaining portion.
- **Expected**: The summarization sub-agent should receive the full tool result as input. If the result is extremely large (e.g., > 100K chars), the summarizer should be pointed to the persisted file on disk (`tools/call-{call_id}.json`) and instructed to read it via its own `read` tool, just as the main agent would.
- **Actual**: `result.output[:20000]` on line 884 of engine.py truncates the summarizer input.

## Gap 4: _CLI_MAPPING sets dead config key for --goal
- **Category**: deviation
- **Section**: §九 配置系统
- **Files**: `myagent/config/loader.py`
- **Description**: The `_CLI_MAPPING` dictionary maps `"goal"` to `("session._goal", None)` (line 58 of loader.py), which sets a `_goal` key under the `session` section of the merged config dict. However, `SessionConfig` has no `_goal` field, so `_dict_to_dataclass` silently ignores this key when converting the dict to `AppConfig`. The actual goal initialization happens correctly in `main.py` via `goal_tracker.set_goal(args.goal)`, making this config path dead code.
- **Expected**: Either remove the `"goal"` entry from `_CLI_MAPPING` (since the goal tracker is initialized directly in main.py), or add a proper `goal` field to `SessionConfig` and route it through the config object.
- **Actual**: Dead config path in `_CLI_MAPPING` that sets a value never consumed.

## Gap 5: context estimation token ratio assumes 3 chars/token for all languages
- **Category**: incomplete
- **Section**: §三 上下文分层 — 压缩策略
- **Files**: `myagent/agent/engine.py`
- **Description**: The `_estimate_context_usage` method uses `total_chars / 3` as its token estimator, commenting "~3 chars per token for mixed Chinese/English". This ratio varies significantly by content type: English text averages ~4 chars/token, code ~3.5 chars/token, and Chinese text can be as low as ~1.5 chars/token. Using a single fixed ratio could over- or under-estimate actual token usage by 30-50%, causing the compression thresholds (75%, 50%, 90%) to fire at incorrect times.
- **Expected**: Use litellm's built-in `token_counter(model, messages)` or an encoder-aware estimator that is content-type sensitive. The LLMProvider class already has a `token_count` method that uses litellm's token counter with a character-based fallback.
- **Actual**: `total_chars / 3` estimation on line 935 of engine.py with no content-type differentiation.

## Gap 6: Dream engine should_run check order may mask the hours condition
- **Category**: incomplete
- **Section**: §六 记忆系统 + 梦境机制 — 触发条件
- **Files**: `myagent/memory/dream.py`
- **Description**: The spec says trigger conditions are "(AND 关系)": both (a) distance > 6h from last dream AND (b) cumulative rounds > 50. The implementation checks `rounds` first and returns False early if rounds are insufficient. While this is semantically correct (both conditions are required), it means the `trigger_hours` check is only evaluated when rounds have already exceeded the threshold. In a long-running session, the elapsed time from the last dream run stored in `last_dream.json` could be stale — the state file is updated at the END of a dream run, not at session boundaries. If a dream run completes mid-session, the timer resets, but if a session ends without triggering a dream, the old timestamp persists. On next startup, if rounds are over threshold, the hours check uses the old timestamp and could trigger too soon.
- **Expected**: The dream state file should be updated at session start (not just dream completion) to reflect that the hours counter resets on each fresh session. Alternatively, track "last session start time" separately from "last dream completion time" and use the minimum of both for the hours check.
- **Actual**: `last_dream.json` is only written at dream completion (lines 160-163 and 518-520 of dream.py). A session that starts, runs many rounds without crossing the dream trigger, then exits, leaves the old timestamp in place.

## Gap 7: context compaction does not expose Layer-3 failure count to the user
- **Category**: incomplete
- **Section**: §三 压缩策略 — 防抖保护
- **Files**: `myagent/context/compression.py`, `myagent/agent/engine.py`
- **Description**: The spec states that after 3 consecutive Layer-3 failures, that layer should be disabled ("降级"). The `CompressionEngine` tracks `_layer3_failures` and skips Layer 3 once the counter reaches 3. However, this degradation is never surfaced to the user or the agent. The user has no way to know that conversation summarization has been disabled and that context pressure may build faster.
- **Expected**: When Layer 3 is disabled after 3 consecutive failures, the engine should emit a notification (`TextChunk`) to inform the user that conversation summarization has been degraded, along with a recommendation (e.g., "use /compact or /clear to manually reduce context").
- **Actual**: `_layer3_failures >= 3` silently disables Layer 3 (line 159 of compression.py) with no user-facing notification.

## Gap 8: SessionStore._write_transcripts writes duration=0 on every save
- **Category**: incomplete
- **Section**: §十 会话系统与项目感知 — 会话结束流程
- **Files**: `myagent/context/persistence.py`
- **Description**: The `_write_transcripts` method writes `"duration": 0` into transcript.json every time it saves (line 506 of persistence.py). The real duration is only calculated and written in `_write_closed_session` at session end. This means that while a session is active, `list_sessions` falls back to computing a live duration (lines 142-149 of persistence.py), and the transcript.json on disk contains an inaccurate `duration: 0` field. If the process crashes or the session is otherwise not cleanly closed, the `duration` field permanently reads 0 even though a live-duration estimate was available at each save point.
- **Expected**: Each call to `_write_transcripts` should compute and write a provisional duration (based on `datetime.now() - session.created_at`) so that crash-recovery always yields a reasonable duration estimate. The `_write_closed_session` call at session end should refine this to the final value.
- **Actual**: `"duration": 0` is hardcoded in the JSON output of `_write_transcripts` (line 506).
