## Blocking / Important Gaps

### G22-01
- **Severity**: Important
- **Design Requirement**: Repository commands and conventions list `ruff check myagent/` as a required lint command (`AGENTS.md:35`, `README.md:143`); lintable Python sources must not include syntactically invalid template placeholders.
- **Actual Implementation**: `myagent/skills/builtin/tdd/templates/test_template.py` is a `.py` file under the `myagent/` lint target but contains raw template placeholders such as `class Test{ClassName}:` and `def test_{method_name}_...`, which are invalid Python syntax. `pyproject.toml` has a `[tool.ruff]` section but no exclude/extend-exclude for this template path (`pyproject.toml:48`).
- **Evidence**: `myagent/skills/builtin/tdd/templates/test_template.py:14`, `myagent/skills/builtin/tdd/templates/test_template.py:28`, `myagent/skills/builtin/tdd/templates/test_template.py:41`, `pyproject.toml:48`.
- **Impact**: `ruff check myagent/` cannot be a reliable required check while an invalid `.py` template is included in the lint target. This is more than low-priority lint style debt because it is a syntax/parsing issue in a first-party path.
- **Suggested Fix**: Rename the template to a non-Python suffix such as `test_template.py.tmpl` or `test_template.py.j2`, or add a precise ruff exclude for `myagent/skills/builtin/**/templates/**`. If the template must ship as a skill resource, update packaging rules at the same time.

### G22-02
- **Severity**: Important
- **Design Requirement**: All exception paths must log `category="error"` with `exception_type`, `traceback`, `context`, and `component` (`AGENTS.md:97`, `AGENTS.md:103`, `docs/superpowers/specs/2026-07-02-myagentcli-design.md:985`). The formatter only emits `exception_type` and `traceback` when `exc_info` is present (`myagent/logging/formatter.py:80`).
- **Actual Implementation**: Several generic exception handlers still return, pass, or log as `system`/`tool` without `exc_info=True`, so structured logs lack required traceback and metadata.
- **Evidence**:
  - `config_set` catches `Exception` and directly returns a tool error without logging (`myagent/tools/builtin/config_tools.py:193`, `myagent/tools/builtin/config_tools.py:194`).
  - `memory_write` catches `Exception` and returns `str(e)` without logging (`myagent/tools/builtin/memory_tools.py:66`, `myagent/tools/builtin/memory_tools.py:67`).
  - task persistence silently swallows load/save failures (`myagent/tools/builtin/session_tools.py:126`, `myagent/tools/builtin/session_tools.py:127`, `myagent/tools/builtin/session_tools.py:139`, `myagent/tools/builtin/session_tools.py:140`).
  - `web_fetch` and `web_search` catch runtime HTTP/API exceptions but log without `exc_info=True` and often as `category="tool"` (`myagent/tools/builtin/web_tools.py:127`, `myagent/tools/builtin/web_tools.py:130`, `myagent/tools/builtin/web_tools.py:279`, `myagent/tools/builtin/web_tools.py:283`, `myagent/tools/builtin/web_tools.py:376`, `myagent/tools/builtin/web_tools.py:380`).
  - MCP client generic capability errors are logged as debug/system instead of error with traceback (`myagent/tools/mcp/client.py:551`, `myagent/tools/mcp/client.py:554`, `myagent/tools/mcp/client.py:581`, `myagent/tools/mcp/client.py:584`), and reader-loop error logs lack `exc_info=True` (`myagent/tools/mcp/client.py:723`, `myagent/tools/mcp/client.py:724`).
  - Agent persistence best-effort errors log as debug/system (`myagent/agent/engine.py:916`, `myagent/agent/engine.py:917`, `myagent/agent/engine.py:1194`, `myagent/agent/engine.py:1195`), and summarization error omits a `context` field (`myagent/agent/engine.py:1012`).
  - Dream still has several `logger.warning(..., exc_info=True, extra={"category": "system"})` exception paths (`myagent/memory/dream.py:1477`, `myagent/memory/dream.py:1550`, `myagent/memory/dream.py:1564`, `myagent/memory/dream.py:1620`, `myagent/memory/dream.py:1647`).
- **Impact**: JSONL logs remain incomplete for audit/debugging. Users can see tool errors, but operators cannot reliably query all exceptions by `category="error"` or recover traceback/context/component.
- **Suggested Fix**: Do a repository-wide exception logging sweep. For unexpected exceptions, use `logger.exception(..., extra={"category": "error", "component": "...", "context": "..."})` or `logger.error(..., exc_info=True, extra=...)`. For deliberately ignored best-effort failures, either justify them as non-exceptional in code comments and use narrow expected exception classes, or log them as error-category with context. Add tests that assert formatter output includes `exception_type`, `traceback`, `component`, and `context` for at least one representative path per module group.
