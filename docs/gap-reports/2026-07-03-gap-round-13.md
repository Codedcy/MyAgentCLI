## Gap 1: [medium] Missing built-in skills SKILL.md files
- **Category**: missing
- **Section**: §七 技能系统 -- 内置技能（计划）
- **Files**: myagent/skills/registry.py, myagent/skills/loader.py
- **Description**: The design spec lists 6 planned built-in skills: brainstorming, code-review, systematic-debugging, tdd, writing-plans, executing-plans. The SkillRegistry scans `myagent/skills/builtin/` for SKILL.md files, but no actual skill directories or SKILL.md files exist in the built-in skills directory. The builtin_dir defaults to `Path(__file__).parent / "builtin"` which is `myagent/skills/builtin/` -- this directory does not exist, so no built-in skills are ever discovered.
- **Expected**: Six skill directories under `myagent/skills/builtin/` each containing a SKILL.md with frontmatter (name, description) and skill instructions, plus optional resources (references/, scripts/, templates/, assets/).
- **Actual**: No built-in skill files exist. The SkillRegistry silently finds zero built-in skills. Only user-level (`~/.myagent/skills/`) and project-level (`.myagent/skills/`) skills can be registered.

## Gap 2: [low] LLM response log retry_count always 0
- **Category**: incomplete
- **Section**: §十一 日志系统 -- LLM 交互记录 (response), §二 LLM 错误处理与重试
- **Files**: myagent/llm/provider.py
- **Description**: The design spec's response log schema includes `"retry_count": <N>` to record how many retries occurred. However, `_complete_with_model()` in `LLMProvider` hardcodes `retry_count=0` in the response log event (line 397: `"retry_count": 0`). When LiteLLM's built-in retry mechanism performs retries internally (up to `litellm.num_retries = 3`), the actual retry count is not captured and recorded. The failure hook (`_register_litellm_failure_hook`) tracks `_failure_count` but this counter is reset per model and never fed into the response log.
- **Expected**: Response log should report the actual number of retry attempts made during the API call. The `_failure_count` counter should be preserved and passed through to the response log event.
- **Actual**: `retry_count` is hardcoded to 0 in every response log entry, even when LiteLLM performed internal retries.

## Gap 3: [medium] spawn_subagent can bypass speculative_exploration config gate
- **Category**: deviation
- **Section**: §八 子 Agent 池与工作流编排 -- background 规则
- **Files**: myagent/tools/builtin/agent_tools.py (SpawnSubagentTool.execute)
- **Description**: The design spec states that for non-Goal mode, background sub-agents are "默认禁止" and require "需显式配置开启" (explicit config to enable, referring to `subagents.speculative_exploration`). The code in `SpawnSubagentTool.execute()` checks `"background" not in params` before forcing background=False:
  ```python
  if not speculative_allowed and "background" not in params:
      background = False
  ```
  This means if the LLM model happens to pass `background=True` explicitly in the tool call params, the config gate is skipped entirely and the sub-agent runs in background mode regardless of the `speculative_exploration` config setting. Since the tool definition declares `"background": {"type": "boolean", "default": true}`, the model will typically include this parameter.
- **Expected**: When `speculative_exploration` is False and no goal is active, `background` should always be forced to False regardless of what the model passes in params. The config gate should not be bypassable by an explicit param.
- **Actual**: The config gate only activates when `background` is absent from params. A model that explicitly passes `background=True` can spawn background sub-agents outside goal mode without config approval.

## Gap 4: [low] Session end permission prompt uses Prompt.ask (prompt_toolkit dependency)
- **Category**: deviation
- **Section**: §十 会话系统 -- 会话结束流程
- **Files**: myagent/agent/session.py (SessionManager.end_session)
- **Description**: The design spec specifies that during session end, the permission persistence prompt should be shown via Rich's `Console.print()` because "此时 prompt_toolkit 已停止" (prompt_toolkit has already stopped at this point). However, the code uses `Prompt.ask()` from `rich.prompt`, which internally depends on prompt_toolkit and may fail or behave unexpectedly if the REPL's PromptSession has already been torn down.
- **Expected**: Use `Console.print()` to display the prompt message, then read user input via a simple method that does not depend on prompt_toolkit (e.g., `sys.stdin.readline()` or Rich's `input()` wrapper that works after prompt_toolkit cleanup).
- **Actual**: `Prompt.ask("是否持久化到配置文件？[Y/n]", choices=["Y", "y", "N", "n"], default="Y")` is called, which uses prompt_toolkit underneath. If the REPL's PromptSession has been torn down or if stdin is in a bad state, this may hang or error.

## Gap 5: [low] Context window uses hardcoded map instead of dynamic model discovery
- **Category**: incomplete
- **Section**: §三 上下文分层结构
- **Files**: myagent/agent/engine.py (_CONTEXT_WINDOW_MAP, _estimate_context_usage)
- **Description**: The design spec says the context window "跟随模型上限" (follows the model's limit). The code uses a hardcoded `_CONTEXT_WINDOW_MAP` dictionary mapping known model names to token limits, defaulting to 1,000,000 for unknown models. This map is not synchronized with LiteLLM's model registry and must be manually maintained. If the configured model's actual context window differs from the hardcoded value (either a new model or an existing model with updated limits), context usage estimates will be inaccurate, potentially triggering premature compression or missing hard truncation thresholds.
- **Expected**: Dynamically query the model's advertised context window through LiteLLM's model info API (e.g., `litellm.model_cost` or the `/v1/models` endpoint), or derive it from the LLM provider's response metadata (e.g., `max_tokens` capability field). Fall back to the hardcoded map only when dynamic discovery fails.
- **Actual**: A static map of ~9 models is maintained in `AgentEngine._CONTEXT_WINDOW_MAP`. Models not in the map default to 1,000,000 tokens.

## Gap 6: [low] Not all log entries include the `category` common field
- **Category**: incomplete
- **Section**: §十一 日志系统 -- 日志事件分类
- **Files**: Multiple files throughout myagent/
- **Description**: The design spec defines `category` as a "公共字段（每条日志都包含）" (common field, every log entry includes it). The JsonLineFormatter correctly extracts `category` from the `extra` dict when present, but numerous log calls throughout the codebase do not pass `category` in their `extra` parameter. These calls produce JSON log entries without the `category` field, violating the spec's assertion that every log entry carries this field.
- **Expected**: Every `logger.info()`, `logger.warning()`, `logger.error()`, and `logger.debug()` call should include `extra={"category": "..."}` with one of the defined category constants (system, llm, tool, agent, subagent, error).
- **Actual**: While major log calls (LLM requests, tool execution, agent iterations, sub-agent lifecycle, errors) properly include `category`, numerous debug-level and some info-level helper logs omit it (e.g., `logger.debug("Failed to persist tool call record", exc_info=True)` in engine.py line 863, `logger.debug("Failed to persist turn to session store", exc_info=True)` in engine.py line 1053).

## Gap 7: [low] `config_set` tool hardcodes boolean key fallback list
- **Category**: incomplete
- **Section**: §九 配置系统
- **Files**: myagent/tools/builtin/config_tools.py
- **Description**: The `ConfigSetTool._validate_value()` method has a fallback hardcoded list of boolean keys (`bool_keys` set) that duplicates information already derivable from the `_TYPE_MAP` (derived from AppConfig dataclass annotations). If new boolean config fields are added to AppConfig but not to the `bool_keys` fallback list, type coercion will be incorrect for those fields.
- **Expected**: The `_TYPE_MAP` derived from dataclass annotations should be the single source of truth for type coercion. The `bool_keys` fallback should be removed or automatically derived from the schema.
- **Actual**: A manual `bool_keys` set at line 226-228 duplicates type information already available in `_TYPE_MAP`. Adding a new boolean field to the schema requires updating both the dataclass AND this hardcoded set, creating a maintenance burden.

## Gap 8: [low] `send_message` tool assigned level 0 but not in spec's level 0 list
- **Category**: deviation
- **Section**: §五 权限/沙箱系统 -- 权限分级
- **Files**: myagent/permissions/controller.py (TOOL_LEVEL_MAP)
- **Description**: The design spec defines permission level 0 as: "read, glob, grep, web_fetch, web_search, task_create, task_update". The code's `TOOL_LEVEL_MAP` maps `send_message` to level 0 as well, with the comment "read-only communication, no filesystem/network side effects." While this classification is reasonable (sending messages has no side effects), `send_message` is not listed in the spec's level 0 tools. This is a deviation from the documented permission level assignment.
- **Expected**: Either add `send_message` to the spec's level 0 list, or assign it to a different level consistent with the spec.
- **Actual**: `send_message` is mapped to level 0 in `TOOL_LEVEL_MAP` but is absent from the spec's level 0 tool enumeration.
