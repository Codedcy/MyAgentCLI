## Gap 1: /exit command requires --force flag (deviation)

- **Category**: deviation
- **Section**: §十 会话系统与项目感知 — 会话结束流程
- **Files**: `myagent/cli/commands.py` (line 228-236)
- **Description**: The `/exit` slash command handler requires `--force` (or `-f`) to set `exit_requested=True`.  Without the flag, the command prints a confirmation message but does NOT trigger the exit flow.  The design spec states: "当用户执行 `/exit`、`/quit` 或 Ctrl+D 退出时，按以下顺序执行" — i.e. `/exit` should directly initiate the session-end sequence.
- **Expected**: `/exit` and `/quit` trigger the session-end flow (stop ReAct loop, call `SessionManager.end_session()`, stop status bar, exit) without requiring a `--force` flag.
- **Actual**: `/exit` without `--force` prints "Goodbye! (Type /exit --force or /quit --force to confirm, or /exit -f)" and returns `exit_requested=False`, keeping the REPL running.

## Gap 2: Post-Layer-3 compression usage estimate is approximate, not measured (incomplete)

- **Category**: incomplete
- **Section**: §三 上下文分层结构 — 压缩策略 (四层渐进)
- **Files**: `myagent/context/compression.py` (line 108-110)
- **Description**: After Layer 3 (conversation summary via LLM call) completes successfully, the compression engine sets `current_usage_pct = self.config.target_after` (default 0.30) as the estimated post-compression usage.  The spec states: "引导目标: ~30%，非硬性" and "达不到30%就接受实际结果" — implying the actual token reduction should be measured and accepted, whatever it is.  The code assumes the reduction always reaches the target rather than measuring it.  Since this value only gates Layer 4 (hard truncation at 90%), and 0.30 < 0.90, Layer 4 never incorrectly fires after Layer 3.  However, the estimate may be misleading if the actual reduction was substantially less.
- **Expected**: After Layer 3, the engine should re-estimate actual context usage (e.g., via `_estimate_context_usage` or character-based token count) and set `current_usage_pct` to the measured value.
- **Actual**: `current_usage_pct` is unconditionally set to `self.config.target_after` (0.30) without measurement.

## Gap 3: 50% context warning flag never resets within a session (incomplete)

- **Category**: incomplete
- **Section**: §三 上下文分层结构 — 压缩策略
- **Files**: `myagent/agent/engine.py` (line 204, 236-248)
- **Description**: The engine uses a boolean flag `context_notified_50` to surface a 50%-usage warning only once per session.  This flag is set to `True` on first crossing of the 0.50 threshold and is never reset.  Per the spec diagram (§三), the 50% notification is part of the normal→warning→auto-compact progression.  If context usage drops well below 50% (e.g., after a successful `/compact` or `/clear`) and later rises above 50% again, the user receives no further warning.  A single notification per session is reasonable to avoid spam, but the spec's flow diagram implies the notification path is always active when usage crosses the threshold from below.
- **Expected**: The flag should reset when context usage drops significantly below 50% (e.g., below 30%), or the notification should be re-emitted if usage re-crosses the 50% threshold upward after previously falling below it.
- **Actual**: `context_notified_50` is set once and never cleared, suppressing all subsequent 50%-crossing warnings for the lifetime of the session.

## Gap 4: Dream sub-agent operation counting relies on regex of natural-language output (incomplete)

- **Category**: incomplete
- **Section**: §六 记忆系统 + 梦境机制
- **Files**: `myagent/memory/dream.py` (line 158-164)
- **Description**: When the dream engine delegates to a sub-agent (`_run_as_subagent`), it counts created/updated/deleted memory operations by applying regex patterns to the sub-agent's NL output text.  The regex searches for English and Chinese keywords like `created? memory`, `updated? memory`, `deleted? memory`.  If the sub-agent uses different phrasing or reports its actions in a narrative style, the count registers as 0 even though memory operations actually occurred.  The spec does not prescribe a counting mechanism, but relying on regex over structured output is inherently fragile.
- **Expected**: The sub-agent should be instructed to produce a structured summary (e.g., a JSON block at the end of its response) with machine-parseable operation counts, or the dream engine should introspect `memory_store.get_session_writes()` after the sub-agent completes.
- **Actual**: Operation counts are extracted from free-text output via regex: `re.findall(r'(?:created?|wrote?|新增|创建)\s+(?:memory|记忆)', analysis_text, re.IGNORECASE)`.

## Gap 5: Session resume does not restore incremental persistence index (incomplete)

- **Category**: incomplete
- **Section**: §十 会话系统与项目感知
- **Files**: `myagent/context/persistence.py` (line 164-190), `myagent/agent/engine.py` (line 1079-1090)
- **Description**: The engine's `_persist_turn` uses a `session._persist_idx` attribute to track which messages have already been persisted to disk, enabling incremental saves.  When a session is resumed via `SessionStore.load_session()`, messages are restored from `transcript.json` but `_persist_idx` is not set.  Consequently, the first call to `_persist_turn` after resume re-persists ALL messages (index 0 through the current end).  This does not cause data loss — persistence is idempotent — but it causes unnecessary disk I/O on every turn after resume.
- **Expected**: `load_session()` should set `session._persist_idx = len(session._messages)` so that subsequent `_persist_turn` calls only persist newly added messages.
- **Actual**: `_persist_idx` is not restored, causing full re-persistence of all messages on the first save after resume.

## Gap 6: /history command shows no tool-call details (incomplete)

- **Category**: incomplete
- **Section**: §一 整体架构 — CLI Layer Slash Commands, §十 会话系统
- **Files**: `myagent/cli/commands.py` (line 192-226)
- **Description**: The `/history` slash command displays recent conversation messages with role and content preview (truncated to 120 chars).  It does not include tool call names, parameters, or result summaries that occurred within the displayed range.  The spec does not prescribe a specific format for `/history`, but the session system (§十) maintains full tool call records in `tools/call-NNN.json` which `/history` could surface to give users a more complete picture of agent activity.
- **Expected**: `/history` could optionally include tool call events (name + brief result) inline with conversation messages.
- **Actual**: `/history` only shows `[ROLE] content_preview` for user and assistant messages; tool calls are invisible.
