## Gap 1: [low] deviation — LLM retry uses custom loop instead of LiteLLM built-in mechanism

- **Category**: deviation
- **Section**: §二 核心 Agent 循环 — LLM 错误处理与重试
- **Files**: `myagent/llm/provider.py`
- **Description**: The design spec states that retryable errors should "使用 LiteLLM 内置 retry 机制". The code implements its own retry loop with `asyncio.sleep` and exponential backoff, never leveraging LiteLLM's built-in retry capabilities. The custom implementation mirrors the same strategy (3 retries, 2s-30s exponential backoff) so behavior is equivalent, but the implementation approach deviates from the spec.
- **Expected**: Litellm's built-in retry is used for retryable errors (as stated in the spec).
- **Actual**: Custom retry loop in `_complete_with_model` with manual exponential backoff via `asyncio.sleep`.

## Gap 2: [low] deviation — Skill content arbitrarily truncated at 4000 characters

- **Category**: deviation
- **Section**: §三 上下文分层结构 / §七 技能系统 — "技能完整内容仅在 invoke 时加载"
- **Files**: `myagent/context/builder.py` (line 302: `content[:4000]`)
- **Description**: The `_format_skill_content` method hard-caps loaded SKILL.md content at 4000 characters: `content = skill.content[:4000]`. The design spec states that when a skill is invoked, the "完整 SKILL.md" should be loaded and injected into the system prompt. This arbitrary truncation violates the spec's intent of loading the full skill instructions. For very long skills, critical instructions could be silently lost.
- **Expected**: Full SKILL.md content is loaded when invoked (spec §七: "加载完整 SKILL.md 注入 system prompt").
- **Actual**: Content truncated to 4000 characters with a "content truncated for context window" message appended.

## Gap 3: [medium] incomplete — TaskUpdateTool missing dependency and metadata fields

- **Category**: incomplete
- **Section**: §四 工具系统 — 内置工具清单 (task_update)
- **Files**: `myagent/tools/builtin/session_tools.py`
- **Description**: The internal `TaskItem` dataclass supports `blocks`, `blocked_by`, `owner`, and `metadata` fields, enabling full task dependency tracking. However, the `TaskUpdateTool` only exposes `taskId`, `status`, `subject`, `description`, and `activeForm` in its parameter schema. This means the LLM cannot set task dependencies or assign owners through the tool, even though the underlying data model supports it. The `task_create` tool also does not expose these fields.
- **Expected**: TaskUpdateTool (and TaskCreateTool) should expose all TaskItem fields as optional parameters: `blocks`, `blockedBy`, `owner`, `metadata`. The spec lists task tracking as a session management tool for the agent to organize complex multi-step work.
- **Actual**: Only `status`, `subject`, `description`, and `activeForm` are exposed for updates. `blocks`, `blockedBy`, `owner`, `metadata` are absent from both tool schemas.

## Gap 4: [low] incomplete — Runtime permission rule adjustment does not support compound patterns

- **Category**: incomplete
- **Section**: §五 权限/沙箱系统 — 对话内调整
- **Files**: `myagent/permissions/controller.py`
- **Description**: The design spec illustrates a compound permission adjustment pattern: "除了 rm -rf 之外都放行" -- which should trigger `set_mode("allow_all")` AND `auto_deny.commands.append("rm -rf")`. The `apply_runtime_rule` method handles individual patterns (single "deny X", single "allow Y", "allow all") but cannot parse the combined "allow all except X" case from a single natural-language command. The model would need to make two separate tool calls or the user would need two separate utterances.
- **Expected**: A single natural-language adjustment like "除了 X 之外都放行" should result in `allow_all` mode plus an auto_deny entry for X (as shown in the spec example).
- **Actual**: `apply_runtime_rule` only handles atomic rules; "除了 rm -rf 之外都放行" would be treated as an unrecognized literal command pattern.

## Gap 5: [low] incomplete — Dream inline fallback uses keyword heuristics instead of LLM analysis

- **Category**: incomplete
- **Section**: §六 记忆系统 + 梦境机制
- **Files**: `myagent/memory/dream.py`
- **Description**: The dream engine has two execution paths: (1) sub-agent-based with full LLM reasoning and (2) inline rule-based fallback. The inline path's contradiction detection (`_detect_contradictions`) uses pre-defined keyword pairs like `("use", "avoid")` and `("always", "never")` to detect contradictions. This is a significant simplification compared to the LLM-driven analysis the spec describes for the dream cycle ("分析" implies semantic reasoning, not simple keyword matching). When no sub-agent pool is available (e.g., minimal installations), dream quality degrades substantially.
- **Expected**: Dream analysis should use semantic reasoning to detect contradictions, new conventions, and stale facts. The spec's dream flowchart shows "分析" as the core step between scanning and writing memories.
- **Actual**: Inline path uses heuristic keyword matching for contradiction detection and basic pattern scanning for transcript analysis. The sub-agent path (preferred) does use LLM reasoning.

## Gap 6: [low] deviation — Config loader has 8 priority levels instead of the specified 7

- **Category**: deviation
- **Section**: §九 配置系统 — 配置优先级（7 层）
- **Files**: `myagent/config/loader.py`
- **Description**: The design spec explicitly defines a 7-level config priority system. The code implements 8 levels because the `--config` CLI flag (custom config path) is treated as an independent file layer (commented as "Level 1.x") between project config and runtime overrides, rather than being folded into the CLI args layer. While the net priority order is correct (explicit CLI flags override custom config), the spec's explicit "7 levels" statement is not matched by the implementation.
- **Expected**: 7 priority levels as documented: CLI args, runtime override, project config, project AGENT.md, user config, user AGENT.md, defaults.
- **Actual**: 8 levels: defaults → user AGENT.md → user config → project AGENT.md → project config → custom config path (`--config`) → runtime overrides → CLI args.
