## Gap 1: [critical] Compression engine is never triggered by the agent loop
- **Category**: missing
- **Section**: 三、上下文分层结构 -- 压缩策略
- **Files**: myagent/agent/engine.py, myagent/context/compression.py
- **Description**: The CompressionEngine implements all 4 compression layers (cleanup, summarise, conversation summary, hard truncation) but the AgentEngine._react_loop never checks token usage or calls compression.compact(). Context compresses only through explicit configuration; the 75% auto-compact trigger and 90% hard-truncation line are dead code.
- **Expected**: At each ReAct iteration the engine should estimate context usage percentage. When >= 75% (configurable), CompressionEngine.compact() should be called. When >= 90%, layer-4 hard truncation must fire.
- **Actual**: CompressionEngine exists and is wired into AgentEngine.__init__ but `self.compression.compact(...)` is never invoked in `_react_loop`. The entire compression pipeline is inert.

## Gap 2: [critical] Permission check is not wired into tool execution in the engine
- **Category**: missing
- **Section**: 五、权限/沙箱系统 -- 确认流程, 四、工具系统 -- 工具调用流程
- **Files**: myagent/agent/engine.py, myagent/tools/builtin/file_tools.py, myagent/tools/builtin/search_tools.py, myagent/tools/builtin/web_tools.py
- **Description**: The PermissionController with its 4-level system (read/write/exec/network_write) and allow/deny lists is fully implemented and wired into AgentEngine.__init__. However, `_execute_tool()` in engine.py never calls `self.permissions.check()` before running a tool. Only BashTool performs its own inline permission check. All other tools (ReadTool, WriteTool, EditTool, GlobTool, GrepTool, WebFetchTool, WebSearchTool, SpawnSubagentTool, SendMessageTool, TaskCreateTool, TaskUpdateTool, MemoryWriteTool) execute unconditionally.
- **Expected**: Engine._execute_tool should call self.permissions.check(tool_name, level, params) before every tool execution. On DENY, return ToolResult with error. On ASK, present confirmation dialog (no timeout). Only on ALLOW proceed to execute.
- **Actual**: PermissionController is constructed in main.py, passed to AgentEngine, but never consulted in the tool execution path. Level 1 (write) and Level 2 (exec) tools run without any permission gate.

## Gap 3: [critical] MCP server startup and tool registration is not connected
- **Category**: missing
- **Section**: 四、工具系统 -- MCP 集成
- **Files**: myagent/cli/main.py, myagent/tools/mcp/client.py, myagent/tools/mcp/adapter.py
- **Description**: MCPClient and MCPToolAdapter are fully implemented with JSON-RPC over stdio, $ref/oneOf schema resolution, stderr drainer, and proper handshake. However, main.py (async_main) never reads mcp.json configuration, never spawns any MCP server subprocesses, and never registers any MCP tools via ToolRegistry. For each configured server, spawn subprocess via MCPClient.start(), call tools/list (and resources/list + prompts/list), wrap each tool in MCPToolAdapter, and register via tool_registry.register(adapter, source="mcp").
- **Actual**: async_main registers only built-in tools. No mcp.json reading. No MCPClient instantiation. No MCP tools registered.

## Gap 4: [critical] Dream engine is never auto-triggered on startup
- **Category**: missing
- **Section**: 六、记忆系统 + 梦境机制 -- 梦境机制
- **Files**: myagent/cli/main.py, myagent/memory/dream.py

## Gap 5: [critical] Compressor references wrong config path for tool_result_max_chars
- **Category**: deviation
- **Section**: 三、上下文分层结构
- **Files**: myagent/context/compression.py
- **Description**: CompressionEngine._layer2_summarize accesses `self.config.tool_result_max_chars` via `hasattr(self.config, 'tool_result_max_chars')`. But self.config is a CompressionConfig dataclass (which has fields: primary_threshold, target_after, hard_limit, minimum_messages, minimum_savings). The field `tool_result_max_chars` lives on ToolsConfig, not CompressionConfig. This means the hasattr check always fails and the fallback value of 5000 is always used -- the user cannot configure the truncation size for layer-2 compression.
- **Expected**: tool_result_max_chars should be read from the ToolsConfig (e.g., via a separate tools_config parameter to CompressionEngine, or by accessing the full AppConfig).
- **Actual**: CompressionEngine.__init__ receives the CompressionConfig subset (via config.context.compression), not the full AppConfig. tool_result_max_chars is invisible to it.

## Gap 6: [high] Stream interruption handling -- no partial content preservation
- **Category**: missing
- **Section**: 二、核心 Agent 循环 -- LLM 错误处理与重试
- **Files**: myagent/agent/engine.py, myagent/llm/provider.py
- **Description**: The design spec requires: "流中断: 流式响应中途断开 → 保留已接收内容 + 注入截断提示，允许用户决定是否继续". When a stream is interrupted, the engine catches the exception and yields an Error event but does NOT preserve the already-received text chunks. The user sees an error message with no option to continue.
- **Expected**: When streaming fails mid-response, collect all received TextChunk content into a buffer, yield a TruncatedText event with the partial content and a prompt like "Stream interrupted. Continue? (y/n)", and let the user decide.
- **Actual**: _react_loop wraps the LLM call in try/except, yields Error(message=str(e)), and returns -- losing all partial text.

## Gap 12: [high] Intent detection incomplete -- only handles "stop" and "continue"
- **Category**: incomplete
- **Section**: 二、核心 Agent 循环 -- 用户交互
- **Files**: myagent/agent/engine.py
- **Description**: The design spec defines four intent signals the model should produce: stop, correct, insert, continue. AgentEngine._detect_intent only handles "stop" and "continue". The "correct" (direction correction) and "insert" (inserting a new task) signals are not detected.
