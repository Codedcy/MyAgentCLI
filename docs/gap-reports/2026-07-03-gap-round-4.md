# Gap Report — Round 4

## Gap 1: [high] retry_callback references undefined status_bar — crash at startup
- **Category**: deviation
- **Section**: §二 核心Agent循环 — LLM 错误处理与重试 ("每次重试时更新状态栏显示重试进度")
- **Files**: `myagent/cli/main.py` (lines 86–89 vs line 177)
- **Description**: The `LLMProvider` constructor receives a `retry_callback` closure that references the local variable `status_bar`. However `status_bar` is first assigned on line 177 — well after `LLMProvider` is constructed (line 83). The expression `if status_bar else None` on line 89 is evaluated eagerly during the call to `LLMProvider(...)`, and since `status_bar` does not exist in the local scope yet, Python raises `UnboundLocalError` at runtime.
- **Expected**: Status bar should receive retry progress updates per spec §二: "每次重试时更新状态栏显示重试进度"
- **Actual**: LLMProvider construction crashes before the REPL starts, or (if the line is reached) the callback is dead code. Retry progress display in the status bar is completely nonfunctional.

## Gap 2: [high] Goal achievement never persisted to session
- **Category**: missing
- **Section**: §二 核心Agent循环 — Goal check ("达成→报告完成"), §十 会话系统
- **Files**: `myagent/agent/engine.py`, `myagent/agent/session.py`, `myagent/context/persistence.py`
- **Description**: When the ReAct loop determines a goal has been achieved, `AgentEngine._react_loop()` yields `Done()` and returns — but never sets `session.goal_achieved = True`. The only place `goal_achieved` is set is in `SessionManager.end_session()` (line 103), which unconditionally sets it to `False` when it is still `None`. This means EVERY session that had a goal will be recorded as "goal not achieved" in transcripts and session listings, even when the goal was in fact completed successfully.
- **Expected**: In the goal-achieved branch of `_react_loop`, before yielding `Done()`, set `session.goal_achieved = True`. At session end, only set to `False` if it is still `None`.
- **Actual**: `session.goal_achieved` is always `False` (or `None`) for all sessions.

## Gap 3: [medium] Session end does not finalize transcript or mark closed
- **Category**: missing
- **Section**: §十 会话系统 — 会话结束流程 step 2a ("标记 session 为 closed，写入 transcript 结束标记")
- **Files**: `myagent/agent/session.py` (line 95–157)
- **Description**: `SessionManager.end_session()` performs permission persistence prompting and memory summary display, but it never calls `self.session_store` to write a final transcript update. The last transcript write occurs during the engine's last `_persist_turn` call, which means: (a) no explicit "session closed" marker is written, (b) `session.goal_achieved` as set in `end_session()` is never persisted to disk, and (c) the final message state may be stale.
- **Expected**: `end_session()` should call `self.session_store._write_transcripts(...)` with the final session state, including a closed marker and the resolved `goal_achieved` value.
- **Actual**: Transcript is last written during the engine loop; end-of-session state changes are never flushed to disk.

## Gap 4: [medium] Dream engine only checks trigger on startup — never mid-session
- **Category**: incomplete
- **Section**: §六 记忆系统 — 梦境机制 ("Agent 启动时，距离上次梦境 > 6 小时 AND 累计对话轮数 > 50 轮")
- **Files**: `myagent/cli/main.py` (line 127), `myagent/memory/dream.py`
- **Description**: The dream engine's `should_run()` is called exactly once during startup. For sessions lasting longer than 6 hours, the dream trigger condition will become true mid-session but will never be checked again. A background task that periodically re-checks the trigger condition is absent.
- **Expected**: While the spec primarily describes startup-time triggering, a long-running REPL session should have a mechanism to detect that the dream interval has elapsed and trigger consolidation mid-session (or at minimum, on next user idle period).
- **Actual**: Dream only runs at startup; long sessions never get a mid-session dream cycle.

## Gap 5: [medium] Status bar token count never updated live
- **Category**: incomplete
- **Section**: §一 CLI Layer — Status Bar ("token 消耗、子Agent 状态、思考模式"); §八 Context 状态栏展示 ("Token: 156K")
- **Files**: `myagent/cli/status.py`, `myagent/cli/repl.py`, `myagent/agent/engine.py`
- **Description**: The `StatusBar` class has a `tokens` data field and `_output_to_console` updates the status bar for sub-agents, but token usage from `Done` events (which carry `usage.total_tokens`) is never propagated to the status bar. The status bar's token display always shows 0 or whatever was last explicitly set.
- **Expected**: After each LLM `Done` event, the total token count should be pushed to the status bar so the user sees live token consumption (e.g., "Token: 156K").
- **Actual**: Status bar token counter is not wired to the engine's token tracking. It stays at 0.

## Gap 8: [low] Error log records lack `context` field per spec
- **Category**: incomplete
- **Section**: §十一 日志系统 — error category ("异常: 所有 except 块必须记录 exception_type + traceback + context（触发时的操作描述）+ component")
- **Files**: `myagent/agent/engine.py` (line 708), `myagent/llm/provider.py`, `myagent/subagent/worker.py`
- **Description**: The design spec requires every error log to include a `context` field describing the triggering operation. Several error log calls omit this field. For example, `engine.py` line 707–710 logs `"Tool '%s' failed: %s"` with `extra={"category": "error", "component": "tool"}` but no `context` key. The formatter handles `exc_info` correctly (producing `exception_type` and `traceback`) but `context` must be explicitly passed in `extra`.
- **Expected**: Error log calls should include `"context"` in `extra` describing the operation that was in progress (e.g., `"read_file", "call_llm", "spawn_subagent"`).
- **Actual**: Many error log sites pass `component` but not `context`.

## Gap 9: [low] Hard truncation (90%) not independently triggered
- **Category**: deviation
- **Section**: §三 上下文分层 — 压缩策略 (flowchart: "90%: 硬上限触发" as a separate path from the 75% automatic compression trigger)
- **Files**: `myagent/agent/engine.py` (lines 207–251), `myagent/context/compression.py` (lines 47–112)
- **Description**: The spec flowchart shows the 90% hard limit as a separate trigger path ("HARSH") independent of the 75% automatic compression path. In the code, Layer 4 truncation is handled inside `compression.compact()`, which is only called when `usage_pct >= 0.75`. If `compact()` returns early due to `minimum_messages` check (e.g., < 10 messages), Layer 4 truncation is skipped entirely — even if usage exceeds 90%. The hard limit should be checked independently.
- **Expected**: A separate check for `usage_pct >= hard_limit` (0.90) that triggers truncation regardless of the 75% compression guard, as specified in the design flowchart.
- **Actual**: Hard truncation is nested inside the 75%-triggered compression flow and can be silently skipped.

## Gap 10: [low] Sub-agent-to-main-agent message channel not implemented
- **Category**: incomplete
- **Section**: §四 工具系统 — send_message ("向子 Agent 或主 Agent 发消息"); §八 — 主 Agent ↔ 子 Agent 交互
- **Files**: `myagent/tools/builtin/agent_tools.py` (SendMessageTool), `myagent/subagent/pool.py`, `myagent/agent/engine.py`
- **Description**: The `send_message` tool's description says it sends messages "to a sub-agent or the main agent." However, the implementation only supports main→sub communication via `pool.send_message(agent_id, message)`. There is no mechanism for a sub-agent to send a message back to the main agent that the main agent's ReAct loop can inspect and act upon. The sub-agent pool has no callback or queue for outbound messages from workers.
- **Expected**: Either (a) implement a sub-to-main messaging channel (e.g., a queue the main agent polls between iterations), or (b) update the tool description/clarify in the spec that the initial version only supports main→sub.
- **Actual**: Only main→sub messaging works. Sub-agent workers have no path to deliver messages to the main agent.

## Gap 11: [low] ContextBuilder.build() receives goal but engine never passes it
- **Category**: deviation
- **Section**: §三 上下文分层 — sending to API format; §二 Goal mode
- **Files**: `myagent/context/builder.py` (line 97: `goal` parameter), `myagent/agent/engine.py` (line 137: `run()` calls `build()` without `goal`)
- **Description**: `ContextBuilder.build()` accepts an optional `goal: str` parameter and injects a `"## Current Goal"` section into the system prompt when it is set. However, `AgentEngine.run()` never passes the current goal to `build()`. The goal injection happens outside the context builder — in `_react_loop()` via `_continue_with_feedback()` — which only adds goal context AFTER a failed goal check, not proactively on every turn.
- **Expected**: On every `build()` call, the engine should extract the current goal from `self.goal_tracker` and pass it as the `goal` parameter, so the goal is always visible in the system prompt.
- **Actual**: The `goal` parameter in `ContextBuilder.build()` is unused by any caller; goal context is only injected reactively after failed goal checks.
