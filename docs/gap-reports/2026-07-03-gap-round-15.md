## Gap 1: L3 project context omits detected environment fields
- **Category**: incomplete
- **Severity**: medium
- **Section**: §三 — 上下文分层结构 (L3: Project Context), §十 — 会话系统与项目感知
- **Files**: `myagent/context/builder.py` (line 264-276, `_format_project_context`), `myagent/agent/project.py` (lines 17-41, `ProjectContext`)
- **Description**: The `ProjectDetector.detect()` method populates `ProjectContext` with `package_manager`, `python_version`, `build_system`, `test_framework`, and `linter` fields. However, `ContextBuilder._format_project_context()` only outputs `project_type`, `git_branch`, `git_status`, `structure_summary`, and `agent_md_content`. The detected environment metadata is available on the object but never included in the L3 system prompt context.
- **Expected**: The spec's L3 "环境感知" (environment awareness) should expose all detected project metadata to the model. Fields like `package_manager` (uv/pip/poetry/npm), `python_version`, `build_system` (make/pyproject), `test_framework` (pytest/unittest), and `linter` (ruff/flake8) are critical for the model to give context-appropriate advice (e.g., suggesting `uv add` instead of `pip install`).
- **Actual**: Only `project_type`, git info, directory structure summary, and AGENT.md/CLAUDE.md content appear in the system prompt. The `package_manager`, `python_version`, `build_system`, `test_framework`, and `linter` fields — while detected — are never surfaced to the LLM.

## Gap 2: Retry backoff parameters not explicitly configured
- **Category**: incomplete
- **Severity**: low
- **Section**: §二 — 核心 Agent 循环 (LLM 错误处理与重试)
- **Files**: `myagent/llm/provider.py` (lines 78-79, 130-134)
- **Description**: The design spec specifies an explicit exponential backoff strategy: "指数退避重试，最多 3 次，初始间隔 2s，上限 30s". The code sets `litellm.num_retries = MAX_RETRIES` (3) and relies on LiteLLM's built-in retry mechanism for the backoff behavior, but does not explicitly configure the base delay (2s) or maximum delay (30s). LiteLLM's default exponential backoff may use different parameters internally.
- **Expected**: LiteLLM's retry configuration should explicitly set `litellm.num_retries = 3` with a `RetryPolicy` or equivalent configuration ensuring the initial delay is 2 seconds and the cap is 30 seconds. Alternatively, the sub-agent's explicit retry loop in `worker.py` (which correctly uses `_SUBAGENT_BASE_DELAY = 2.0` and `_SUBAGENT_MAX_DELAY = 30.0`) could serve as the reference implementation pattern.
- **Actual**: Only `litellm.num_retries = 3` is set. The base delay and maximum delay are left to LiteLLM's internal defaults, which are not explicitly documented as matching the 2s/30s values required by the spec.

## Gap 3: Custom TimedSizeRotatingFileHandler diverges from spec's stated implementation approach
- **Category**: deviation
- **Severity**: low
- **Section**: §十一 — 日志系统 (实现概览)
- **Files**: `myagent/logging/logger.py` (lines 47-118, `TimedSizeRotatingFileHandler`)
- **Description**: The design spec's implementation overview states: "日志文件通过 `logging.handlers.TimedRotatingFileHandler` 按天轮转 + `RotatingFileHandler` 按大小轮转组合". The code implements a custom single-class `TimedSizeRotatingFileHandler` that subclasses `RotatingFileHandler` and adds date-based rollover internally, rather than composing the two standard-library handlers as described.
- **Expected**: Two separate handlers from the standard library — `logging.handlers.TimedRotatingFileHandler` for daily rotation and `logging.handlers.RotatingFileHandler` for size-based rotation — combined in the QueueListener handler list. This is the approach documented in the spec.
- **Actual**: A custom `TimedSizeRotatingFileHandler` class subclasses `RotatingFileHandler` and adds midnight rollover by checking the date on each `emit()` call and manually reopening the file stream when the day changes. Functional outcome is equivalent but the implementation approach differs from what the spec documents.

## Gap 4: Dream inline fallback path cannot correct single-memory factual errors
- **Category**: incomplete
- **Severity**: low
- **Section**: §六 — 记忆系统 + 梦境机制 (梦境机制 flowchart: "纠正错误记忆")
- **Files**: `myagent/memory/dream.py` (lines 359-554, `_run_inline`)
- **Description**: The design spec's dream flowchart includes "纠正错误记忆" (correct erroneous memories) as one of the four analysis outputs. The sub-agent path (`_run_as_subagent`) can detect and correct factual errors within individual memory files through LLM reasoning. However, the inline fallback path (`_run_inline`) only handles: (a) empty/placeholder memory cleanup, (b) description-based deduplication, (c) contradiction detection between pairs of memories (merge by keeping newer), and (d) staleness detection. There is no mechanism in the inline path to identify and correct factual errors within a single, non-contradictory memory file.
- **Expected**: The inline dream path should include a mechanism to detect potentially incorrect facts within individual memory files. While full LLM-based correction requires the sub-agent path, the inline path could at minimum flag memories whose content contradicts project conventions detected during startup (e.g., a memory saying "use camelCase" when the project uses snake_case) or memories that contain obviously invalid metadata.
- **Actual**: The inline path can only handle contradictions between pairs of memories (one asserts X, another asserts not-X). A single memory containing an incorrect fact (e.g., "the project uses Python 3.8" when the detected version is 3.12) would not be caught.
