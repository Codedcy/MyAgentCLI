## Gap 1: [high] spawn_subagent speculative_exploration guard cannot detect goal mode
- **Category**: incomplete
- **Section**: §八 子 Agent 池与工作流编排 — background rules
- **Files**: `myagent/tools/builtin/agent_tools.py:40-43`
- **Description**: The `SpawnSubagentTool.execute()` method attempts to detect whether the session is in goal mode by checking `getattr(context.config, '_goal', None)` or `getattr(context.config.session, '_goal', None)`. Neither `AppConfig` nor `SessionConfig` has a `_goal` attribute. The goal is tracked exclusively by `GoalTracker`, which is owned by `AgentEngine`, not stored in config. As a result, the `goal` variable is always `None`, causing the speculative_exploration guard to treat every session as non-goal mode. This means background sub-agents in goal mode are incorrectly blocked unless `speculative_exploration` is explicitly enabled in config.
- **Expected**: The goal status should be obtained from `GoalTracker` (e.g., via a new field on `ToolContext` that holds a reference to the goal tracker, or by checking `GoalTracker.get_goal()`).
- **Actual**: The code dereferences `context.config._goal` and `context.config.session._goal` — attributes that do not exist on the configuration dataclasses. The check is effectively dead code; goal mode is never detected.

## Gap 2: [medium] --config CLI argument accepted but not wired to ConfigLoader
- **Category**: missing
- **Section**: §九 配置系统 (7-level priority, CLI start args at level 1)
- **Files**: `myagent/cli/main.py:35,47-49`, `myagent/config/loader.py:128-136`
- **Description**: The `parse_args()` function defines `--config` for a custom config file path, but `async_main()` never reads `args.config` or passes it anywhere. `ConfigLoader.__init__()` accepts only `project_dir` and `user_home` — there is no parameter for a custom config path. The flag is parsed and silently discarded.
- **Expected**: `--config <path>` should allow the user to specify an alternative YAML config file. `ConfigLoader` should accept an optional config path and load it as one of the merge layers, or as a replacement for the user-level config.
- **Actual**: The `--config` flag is a no-op. The argument is defined in `parse_args()` but never consumed.

## Gap 3: [medium] SubAgentWorker transcript collection fields never populated
- **Category**: incomplete
- **Section**: §八 子 Agent 池 — transcript persistence (also §三 持久化策略 subagents/ layout)
- **Files**: `myagent/subagent/worker.py:61-62,88-256`, `myagent/subagent/pool.py:61-113`
- **Description**: `SubAgentWorker` declares `_transcript_messages: list[dict]` and `_transcript_tool_calls: list[dict]` (lines 61-62) but `_run_impl()` never appends to either list during execution. The `_persist_subagent_transcript()` function in pool.py reads these lists to build the sub-agent transcript JSON/Markdown, but they are always empty. This means sub-agent transcript files (`subagents/sub-NNN/transcript.{json,md}`) contain zero messages and zero tool calls.
- **Expected**: After each LLM response and tool execution, `_transcript_messages` and `_transcript_tool_calls` should be populated with the collected data so that `_persist_subagent_transcript` can write complete transcripts.
- **Actual**: Both lists remain `[]` throughout the worker's lifecycle. Transcript persistence writes files with empty message and tool_call arrays.

## Gap 4: [medium] SessionStore base_dir ignores config.session.sessions_dir
- **Category**: incomplete
- **Section**: §九 配置系统 (session section) + §十 会话系统
- **Files**: `myagent/cli/main.py:98`, `myagent/config/schema.py:82`, `myagent/context/persistence.py:56-57`
- **Description**: `SessionConfig.sessions_dir` is defined as `"~/.myagent/sessions/"` in the config schema, and the design spec shows this as a user-configurable path. However, `SessionStore` is constructed with `SessionStore()` (no arguments) in `main.py`, and its `__init__` always defaults to `Path.home() / ".myagent" / "sessions"`. The config value for `sessions_dir` is never read or passed to `SessionStore`.
- **Expected**: `SessionStore` should accept a `base_dir` from the config (or be constructed with the resolved `config.session.sessions_dir`), so users can change where session data is stored.
- **Actual**: `SessionStore(base_dir)` defaults to `~/.myagent/sessions/` hardcoded; the `sessions_dir` config key is never consumed.

## Gap 5: [low] Edit tool does not enforce "must read file first" constraint
- **Category**: incomplete
- **Section**: §四 工具系统 — edit tool description
- **Files**: `myagent/tools/builtin/file_tools.py:120-178`
- **Description**: The design spec description for the `edit` tool states: "You must Read the file in this conversation before editing, or the call will fail." The current `EditTool.execute()` performs no such check — it will edit any file regardless of whether it was previously read in the session. The `WriteTool` has `_read_files` tracking (though it does not block writes either), but `EditTool` has no tracking or enforcement at all.
- **Expected**: `EditTool` should track which files have been read in the current session and refuse to edit files that were never read, returning an informative error.
- **Actual**: `EditTool.execute()` edits any file at any path without any read-history check.

## Gap 6: [low] Write tool does not enforce "must read file before overwriting" constraint
- **Category**: incomplete
- **Section**: §四 工具系统 — write tool description
- **Files**: `myagent/tools/builtin/file_tools.py:78-117`
- **Description**: The design spec description for the `write` tool states: "Overwriting an existing file you haven't Read will fail." The `WriteTool` has a `_read_files` class-level set that tracks read files, and it adds the path to the set on overwrite — but it never actually blocks a write based on the set. The `path.exists() and str(path) not in self._read_files` check only adds the path to the set; it does not return an error.
- **Expected**: When `path.exists()` and `str(path) not in self._read_files`, `WriteTool` should return `ToolResult(error="Must read file before overwriting: ...")`. Only after the file has been read should overwrites be allowed.
- **Actual**: The `_read_files` set is populated but never used as a guard condition. Any file can be overwritten at any time.

## Gap 7: [low] Dream engine "correction → memory" uses fragile string matching
- **Category**: incomplete
- **Section**: §六 记忆系统 — 梦境机制 (反复纠正 → 整合为记忆)
- **Files**: `myagent/memory/dream.py:478-541`
- **Description**: The design spec states that the dream engine should detect "反复纠正 ≥2次 → 整合为记忆" (repeated corrections ≥2 times → consolidate into memory). The code detects correction patterns in `_scan_transcripts()` and reports them as findings. However, `_create_memories_from_patterns()` only creates the "common-corrections" memory when the finding text literally contains both "corrections" AND "detected". This is a fragile substring match — if the finding message wording changes even slightly (e.g., the LLM analyzer generates a different phrasing), the memory will not be created. Additionally, the created memory is a generic placeholder rather than an analysis of what specific corrections recurred.
- **Expected**: The dream engine should create memories based on the actual count (`correction_count >= 2`), not on substring matching of the finding text. The created memory should describe the specific correction patterns found (e.g., "The agent was repeatedly corrected about using pytest for testing").
- **Actual**: Memory creation depends on `/corrections.*detected/i` substring matching against finding text. The created memory is a generic template: "This project has patterns of repeated corrections across sessions."

## Gap 8: [low] config_set tool not in the design spec tool list
- **Category**: deviation
- **Section**: §四 工具系统 — 内置工具清单
- **Files**: `myagent/tools/builtin/config_tools.py`
- **Description**: The `ConfigSetTool` (`config_set`) is registered as a built-in tool (line 301 of `main.py`) but does not appear in the design spec's built-in tool list (which enumerates 13 tools: read, write, edit, glob, grep, bash, spawn_subagent, send_message, task_create, task_update, memory_write, web_fetch, web_search). While this tool implements useful runtime config changes consistent with §九's "运行时覆盖" layer, it is an addition beyond the designed tool surface.
- **Expected**: Either the tool should be in the spec, or it should not be registered as a built-in (deferred to a later phase per the implementation plan).
- **Actual**: 14 built-in tools are registered, one more than the 13 listed in the design spec.

## Gap 9: [low] send_message permission level 2 not documented in design spec levels
- **Category**: deviation
- **Section**: §五 权限系统 — 权限分级
- **Files**: `myagent/permissions/controller.py:45`
- **Description**: The permission level mapping assigns `send_message` to level 2 (exec), alongside `bash` and `spawn_subagent`. However, the design spec's permission level table does not include `send_message`. Level 2 is defined as including `bash` and `spawn_subagent` only.
- **Expected**: `send_message` should either be added to the spec's level table (e.g., level 2 or a more appropriate level) or it should not appear in TOOL_LEVEL_MAP, defaulting to level 3 (which is the fallback for unknown tools).
- **Actual**: `send_message` is mapped to level 2 without spec coverage.

## Gap 10: [low] MCP resources and prompts are fetched but not integrated into the context system
- **Category**: incomplete
- **Section**: §四 工具系统 — MCP 集成 (resources/list, prompts/list)
- **Files**: `myagent/cli/main.py:400-415`
- **Description**: The design spec's MCP integration sequence diagram shows `resources/list` and `prompts/list` being fetched at startup. The code correctly calls both methods (`client.list_resources()` and `client.list_prompts()`) and logs the results. However, the discovered MCP resources and prompts are never integrated into the agent's context or tool system. Resources could be useful as available reference data; prompts could be registered as callable templates. The current implementation fetches them but discards the data.
- **Expected**: MCP resources should be made available to the context builder (e.g., as discoverable assets the LLM can request). MCP prompts should be registered as callable prompt templates the agent can invoke.
- **Actual**: Resources and prompts are logged at INFO/DEBUG level and then discarded. They are never stored, indexed, or exposed to the agent.
