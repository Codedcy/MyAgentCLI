## Gap 1: [critical] /skill-name forced invocation not implemented

- **Category**: missing
- **Section**: 七、技能系统 — 调用方式
- **Files**: myagent/cli/commands.py, myagent/cli/repl.py
- **Description**: When the user types `/skill-name` (如 `/code-review`), CommandDispatcher does not route to the skill registry. It only handles 7 hardcoded slash commands (`/mode`, `/goal`, `/skills`, `/dream`, `/clear`, `/history`, `/exit`) and returns "Unknown command" for anything else.
- **Expected**: Per design spec: "强制调用: `/skill-name` 前缀，强制加载并执行指定技能". The REPL should recognize skill names from the skill registry and trigger `skill_invoke` to load the full SKILL.md into context.
- **Actual**: All unrecognized `/xxx` inputs return `CommandResult(output=f"Unknown command: /{cmd_name}", success=False)`.

## Gap 2: [high] Dream engine does not create or update memories from analysis

- **Category**: incomplete
- **Section**: 六、记忆系统 + 梦境机制 — 梦境机制
- **Files**: myagent/memory/dream.py
- **Description**: The dream engine runs analysis (scanning transcripts for correction patterns, detecting contradictions, identifying stale memories) but only writes findings to the dream log. It never creates new memory files or updates existing ones based on its analysis, nor does it spawn an LLM-powered sub-agent to consolidate knowledge.
- **Expected**: Per spec flowchart:
  - A1 "新约定? → 写入记忆"
  - A2 "反复纠正? ≥2次 → 整合为记忆"
  - A4 "矛盾记忆? → 合并去重"
  - Then "UPDATE[更新 memory/ 文件 + MEMORY.md]"
  The dream engine should spawn a background sub-agent to analyze findings and create/update/merge memory files.
- **Actual**: DreamResult.memories_created and memories_updated always remain 0. Only deletions (empty memories, duplicates) are performed. The dream report describes findings narratively but takes no action on them.

## Gap 3: [high] MCP prompts/list not called during startup

- **Category**: missing
- **Section**: 四、工具系统 — MCP 集成
- **Files**: myagent/cli/main.py (_startup_mcp_servers)
- **Description**: The MCP server startup flow in `_start_single_mcp_server` calls `client.list_tools()` and `client.list_resources()`, but never calls `client.list_prompts()` or the equivalent `prompts/list` MCP method.
- **Expected**: Per spec sequence diagram: "Startup->>Proc: resources/list + prompts/list". Both resources and prompts should be enumerated during MCP server initialization.
- **Actual**: Only `client.list_resources()` is called. MCPClient has no `list_prompts` method at all.

## Gap 4: [medium] Permission config persistence is a placeholder (not wired)

- **Category**: incomplete
- **Section**: 五、权限/沙箱系统 — 对话内调整
- **Files**: myagent/agent/session.py (end_session)
- **Description**: At session end, the user is prompted to persist permission rule changes made during the conversation. However, the actual YAML file write to `~/.myagent/config.yaml` or `.myagent/config.yaml` is not implemented — a placeholder message is shown: "持久化配置文件写入功能在 v1.1+ 支持".
- **Expected**: Write updated permission rules (auto_allow/auto_deny changes) to the appropriate config YAML file. The spec says: "会话结束后询问是否持久化到配置文件" with actual file persistence.
- **Actual**: Rich prompt dialog appears but only a placeholder message is printed. The permission changes collected via `permissions.get_session_changes()` are never written to disk.

## Gap 5: [medium] No auto-completion (Completer) in REPL

- **Category**: missing
- **Section**: 一、整体架构 — CLI Layer (REPL Engine)
- **Files**: myagent/cli/repl.py
- **Description**: prompt_toolkit `PromptSession` is created without a `completer` parameter. The design spec's architecture table lists "自动补全" (auto-completion) as a feature of the REPL Engine.
- **Expected**: prompt_toolkit with `WordCompleter` (for slash commands, mode values, skill names) or a custom `Completer` implementation that suggests file paths and command names.
- **Actual**: `PromptSession(history=FileHistory(...), multiline=True, key_bindings=kb)` — no `completer` argument.

## Gap 6: [medium] LLM response prompt files not written to .prompts/

- **Category**: incomplete
- **Section**: 十一、日志系统 — LLM 交互记录
- **Files**: myagent/llm/provider.py (_write_prompt_logs)
- **Description**: When `logging.llm_prompts=true` and log level is DEBUG, only request JSON files are written to `~/.myagent/logs/.prompts/`. The spec also requires response JSON files for each API call.
- **Expected**: Per spec, two files per API call:
  ```
  .prompts/2026-07-03T14-32-01-abc123-request.json    ← 完整 messages + tools
  .prompts/2026-07-03T14-32-04-abc123-response.json   ← 完整 response
  ```
- **Actual**: Only request files are written. `_write_prompt_logs` is called before the API call; there is no corresponding call after the response is received.

## Gap 7: [medium] StatusBar runs its own Live display, uncoordinated with REPL

- **Category**: incomplete
- **Section**: 一、整体架构 — CLI Layer (Status Bar)
- **Files**: myagent/cli/status.py, myagent/cli/repl.py
- **Description**: StatusBar creates its own Rich `Live` instance via `Live(layout, ...).start()`, and the REPL creates its own `Console()` independently for streaming output. These are two separate Rich rendering contexts — they may overlap, hide each other, or produce garbled output.
- **Expected**: The status bar should be a persistent top region in a shared Rich `Layout`, with streaming content rendered below it. The spec shows a unified display where the status bar is always visible while messages stream underneath.
- **Actual**: `StatusBar.start()` creates a standalone `Live(layout, refresh_per_second=4)` with a split layout ("top" size=3, "main"). The REPL's `Console.print()` writes to an entirely separate output stream with no awareness of the status bar's display area.

## Gap 8: [medium] StatusBar sub-agent details show only raw status enum values

- **Category**: incomplete
- **Section**: 八、子 Agent 池 — Context 状态栏展示
- **Files**: myagent/cli/status.py, myagent/cli/main.py
- **Description**: The status bar's sub-agent detail lines show only raw AgentStatus enum values (e.g., `sub-001: running`). The design spec shows a much richer display with task names, progress percentages, result summaries, and retry counts with emoji indicators.
- **Expected**: Per spec example:
  ```
  ├─ review-auth      ⏳ 审查中... (82%)
  ├─ review-api       ✅ 完成 (2 个问题)
  └─ review-middleware 🔄 重试中 (1/3)
  ```
- **Actual**: Shows only `f"{hid}: {h.status.value}"` — raw agent IDs and status enum names. No task names, no progress, no result summaries.

## Gap 9: [medium] Compression minimum_savings check never enforced

- **Category**: incomplete
- **Section**: 三、上下文分层 — 压缩策略（四层渐进）防抖保护
- **Files**: myagent/context/compression.py
- **Description**: The `CompressionConfig.minimum_savings` field (default 0.10) is defined in the schema but never read or enforced in the compression engine. The spec's debounce rule states: "压缩后 token 减少 < 10% → 跳过".
- **Expected**: After Layer 2 summarization, compare the estimated token reduction; if savings are below `minimum_savings` (10%), skip further compression layers for this cycle.
- **Actual**: Layer 2 always applies when above the primary threshold, regardless of actual savings. The `minimum_savings` config field exists in the schema but is dead code.

## Gap 13: [low] Dream engine detects stale memories but does not delete them

- **Category**: incomplete
- **Section**: 六、记忆系统 + 梦境机制 — 梦境机制 (A3)
- **Files**: myagent/memory/dream.py
- **Description**: Stale memories (not accessed in > 30 days) are identified and listed in the dream report, but they are never actually deleted. The `run()` method only deletes empty memories (< 20 chars) and duplicates.
- **Expected**: Per spec flowchart A3: "过时记忆? → 删除". The dream engine should delete stale memory files, not just report them.

## Gap 14: [low] ContextBuilder internal variable naming deviates from 6-layer model

- **Category**: deviation
- **Section**: 三、上下文分层 — 六层模型
- **Files**: myagent/context/builder.py (build method)
- **Description**: The `build()` method uses internal variables `l5` for active skill content and `l6` for goal context. In the design spec's model, L5 is conversation history and L6 is current input. While the final assembled system prompt includes all correct content, the variable naming creates confusion and misalignment with the documented layer model.
- **Expected**: Layer identifiers should match the spec: L0=system prompt, L1=tools, L2=skills index, L3=project, L4=memory, L5=conversation, L6=current input. Active skill content and goal context are not independent layers — they are injected into the appropriate sections.
- **Actual**: `l5 = self._format_skill_content(skill)` and `l6 = f"## Current Goal\n{goal}"`. These are not actual context layers, creating a mismatch between code and documentation.
