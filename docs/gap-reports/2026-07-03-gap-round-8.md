## Gap 1: [medium] Sub-agent LLM call has no retry logic
- **Category**: missing
- **Section**: §二 核心 Agent 循环 — LLM 错误处理与重试
- **Files**: myagent/subagent/worker.py
- **Description**: SubAgentWorker._run_impl() (line 191-198) catches all LLM exceptions and immediately returns `"Error: {e}"` without any retry. The design spec requires sub-agents to share the same retry strategy (exponential backoff, max 3 retries, 2s initial / 30s max) with silent retries for sub-agents.
- **Expected**: Sub-agent LLM calls should retry up to 3 times with exponential backoff, identical to the main agent's retry strategy. Retries should be silent (not notified to user), and only the final failure should be reported back to the main agent.
- **Actual**: SubAgentWorker catches all exceptions from `self.llm.complete()` once and immediately returns `f"Error: {e}"`.

## Gap 2: [medium] Widespread missing traceback in error logging
- **Category**: incomplete
- **Section**: §十一 日志系统 — 异常日志要求
- **Files**: myagent/agent/engine.py, myagent/subagent/worker.py, myagent/subagent/pool.py, myagent/tools/mcp/client.py
- **Description**: Multiple `except` blocks use `logger.error()` without `exc_info=True`, which means they fail to capture `exception_type` and `traceback` in the log record, despite the spec requiring all except blocks to record `exception_type + traceback + context`.
- **Expected**: Every `except` block in the codebase should log with either `logger.exception()` or `logger.error(..., exc_info=True)` to ensure full traceback is captured. The spec explicitly states: "所有 except 块必须记录 exception_type + traceback + context（触发时的操作描述）+ component（llm/tool/agent/mcp）".
- **Actual**: Specific instances without traceback:
  - `engine.py:337`: `logger.error("LLM error in iteration %d: %s", ...)` — stream interruption
  - `engine.py:793`: `logger.error("Tool '%s' failed: %s", ...)` — tool execution failure
  - `worker.py:254`: `logger.error("Tool '%s' failed: %s", ...)` — sub-agent tool failure
  - `pool.py:369`: `logger.error("Sub-agent %s failed: %s", ...)` — sub-agent run failure
  - `mcp/client.py:342`: `logger.error("MCP reader loop error: %s", ...)` — MCP protocol error

## Gap 3: [medium] "3 consecutive retries exhausted" produces raw error instead of user guidance
- **Category**: incomplete
- **Section**: §二 核心 Agent 循环 — LLM 错误处理与重试
- **Files**: myagent/agent/engine.py
- **Description**: When LLMProvider exhausts all retries (3 attempts) across all fallback models, it raises an `LLMError`. Engine catches this at line 333 and yields `Error(message=str(e))`. The spec says the user should see specific guidance: "降级提示用户检查网络/API key".
- **Expected**: After 3 consecutive retries fail, the engine should produce a user-facing message specifically advising to check network connectivity and API key validity, not just the raw exception text.
- **Actual**: The engine yields a generic `Error(message=str(e))` where `str(e)` is the LLMError's raw message (e.g., "LLMError(rate_limit): ..." or "all_models_exhausted").

## Gap 4: [medium] JSONL log files are never cleaned up by retention policy
- **Category**: incomplete
- **Section**: §十一 日志系统 — 日志目录与轮转
- **Files**: myagent/logging/logger.py
- **Description**: `LogManager._cleanup_old_logs()` (line 266-273) only globs for `myagent*.log*`, which matches `.log` files but NOT `.jsonl` files. Since JSONL is the default log format (`format: jsonl`), the primary log files are never cleaned up by the retention policy, causing unbounded disk growth.
- **Expected**: The cleanup glob should cover all log file extensions, e.g., `myagent*.*` or explicitly include both `myagent*.jsonl*` and `myagent*.log*`.
- **Actual**: Only `myagent*.log*` files are cleaned up. `myagent-YYYY-MM-DD.jsonl` and its rotated siblings (`myagent-YYYY-MM-DD.1.jsonl`) accumulate indefinitely.

## Gap 5: [low] Ctrl+C on idle resets buffer instead of prompting "Exit? (y/n)"
- **Category**: deviation
- **Section**: §十 会话系统 — 会话结束流程
- **Files**: myagent/cli/repl.py
- **Description**: The prompt_toolkit key binding for Ctrl+C (line 175-186) resets the input buffer when the engine is idle, rather than triggering the "Exit? (y/n)" confirmation flow. The spec states: "Ctrl+C 在 idle 状态时触发同样的退出流程（先确认 'Exit? (y/n)'）".
- **Expected**: When no engine task is running, Ctrl+C should trigger the exit confirmation dialog ("Exit? (y/n)"). If confirmed, the session ends via the normal shutdown flow (permission persistence prompt, memory summary, goodbye).
- **Actual**: Ctrl+C on idle resets the buffer to empty without any exit prompt. The "Exit? (y/n)" dialog only triggers via the outer KeyboardInterrupt handler (line 204), which never fires because the key binding consumes the event.

## Gap 7: [low] Retry progress info set on status bar is never displayed
- **Category**: incomplete
- **Section**: §二 核心 Agent 循环, §八 Context 状态栏展示
- **Files**: myagent/cli/status.py, myagent/cli/main.py
- **Description**: Main.py wires a `retry_callback` (line 91-93) that calls `status_bar.update(retry_info=...)` during LLM retries. However, `StatusBar.get_renderable()` never reads the `retry_info` key from `self._data`, so the retry progress is silently discarded and never shown to the user.

## Gap 8: [low] Session directory path resolution uses string replace instead of expanduser
- **Category**: deviation
- **Section**: §十 会话系统 — 环境感知
- **Files**: myagent/cli/main.py
- **Description**: Line 99-102 resolves `config.session.sessions_dir` using `str.replace("~", str(Path.home()))`. This is a fragile string-based approach that could mangle paths containing literal `~` characters in non-home-directory positions. The ConfigLoader itself uses proper regex-based `~` expansion (see loader.py line 308-313).
- **Expected**: Use `Path(sessions_dir_raw).expanduser()` which correctly expands only a leading `~` or `~user` prefix.
- **Actual**: `sessions_dir_raw.replace("~", str(Path.home()))` which is a blind string substitution.

## Gap 10: [low] Open sessions always show duration 0 in listing
- **Category**: incomplete
- **Section**: §十 会话系统 — 会话列表展示
- **Files**: myagent/context/persistence.py
- **Description**: `_write_transcripts()` always sets `"duration": 0` in transcript.json. Duration is only computed in `_write_closed_session()` by comparing `closed_at` to `created_at`. Sessions listed while still open will show `duration: 0` (displayed as "—" in the listing output).
- **Expected**: For open sessions, compute the duration at read time from `created_at` to `now()`, or update the duration periodically during the session.
- **Actual**: Open sessions always have `"duration": 0` in the transcript, which renders as "—" in session listings.

- **Category**: incomplete
- **Section**: §四 工具系统 — 内置工具清单
- **Files**: myagent/tools/builtin/web_tools.py
- **Description**: When the DuckDuckGo Instant Answer API is unavailable (line 223-230), `WebSearchTool` returns a stub message like `"Web search for: {query}\n\n(DuckDuckGo API unavailable — {e})"`. ## Gap 13: [low] LLM request logging doesn't always include 'event' field in extra
- **Category**: incomplete
- **Section**: §十一 日志系统 — LLM 交互记录
- **Files**: myagent/llm/provider.py
- **Description**: The spec defines request and response logging with explicit `event: "request"` and `event: "response"` fields. The provider at line 239-254 includes `"event": "request"` in extra, and at line 285-300 includes `"event": "response"`. However, the LogManager startup log at logger.py line 232-243 uses a nested `extra_fields` dict pattern (`extra={"extra_fields": {...}}`) while the provider uses the flat pattern (`extra={"category": "llm", "event": "request", ...}`). Both patterns are supported by the formatter but the inconsistency could cause confusion. This is minor.
- **Expected**: Consistent use of the flat `extra={...}` pattern across all logging calls.
- **Actual**: Some log calls use the legacy `extra={"extra_fields": {...}}` pattern (LogManager startup) while others use the flat pattern (LLMProvider, AgentEngine).

## Gap 14: [low] SendMessageTool 'main' target path has incomplete wiring
- **Category**: incomplete
- **Section**: §八 子 Agent 池与工作流编排 — 主 Agent ↔ 子 Agent 交互
- **Files**: myagent/tools/builtin/agent_tools.py, myagent/subagent/worker.py, myagent/agent/engine.py
- **Description**: The spec's sequence diagram shows sub-agents sending messages back to the main agent via `send_message`. The `SubAgentPool.send_to_main()` and `drain_outbound_messages()` provide this channel, and the engine drains messages between ReAct iterations (engine.py line 195-208). However, sub-agent workers do not have access to the pool's `send_to_main` method — the worker only has `tool_context` but not a reference to the pool. The `SendMessageTool.execute()` at line 105-111 checks for `target == "main"` and calls `pool.send_to_main()`, but this tool is invoked by the sub-agent worker, which does not have `pool` in its accessible context.
- **Expected**: Sub-agent workers should have a direct reference to the pool (or a callback) to send messages to the main agent. The `ToolContext` in the worker (line 231-236) is set with `pool` available, so this may work if `tool_context.subagent_pool` is properly set when the sub-agent calls tools.
- **Actual**: The sub-agent constructs its own `ToolContext` at worker.py line 231-236: `ctx = self.tool_context or ToolContext(...)`. The `self.tool_context` is passed from the pool and includes `subagent_pool`. So the wiring may work, but the sub-agent's constructed fallback `ToolContext` at line 231 has `subagent_pool` defaulting to None. If `self.tool_context` is properly set, this works; if not, send_message to "main" silently fails.
