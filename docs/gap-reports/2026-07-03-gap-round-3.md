## Gap 1: [critical] Dream engine calls MemoryStore.write() with wrong keyword arguments — runtime crash

- **Category**: deviation
- **Section**: 六、记忆系统 + 梦境机制
- **Files**: `myagent/memory/dream.py` (lines 437-441, 482-490, 503-511), `myagent/memory/store.py` (line 58)
- **Description**: The dream engine's `_merge_contradictions()` and `_create_memories_from_patterns()` call `self.memory_store.write(name=..., content=..., description=...)`, but `MemoryStore.write()` signature is `write(self, file_path: str, content: str)`. The keyword arguments `name` and `description` do not exist on the method, so these calls will raise `TypeError` at runtime.
- **Expected**: Dream engine should construct a proper `file_path` (e.g., `str(memory_dir / f"{name}.md")`) and pass it as the first positional argument, or MemoryStore.write() should accept optional keyword arguments for name/description with a path-construction fallback.
- **Actual**: `await self.memory_store.write(name="common-corrections", content=..., description=...)` — invalid signature.

## Gap 2: [high] Log category fields not serialized — JsonLineFormatter expects `extra_fields` but most code uses standard `extra`

- **Category**: deviation
- **Section**: 十一、日志系统
- **Files**: `myagent/logging/formatter.py` (lines 42-49), `myagent/agent/engine.py` (lines 177-180, 198-201, etc.), `myagent/llm/provider.py` (lines 239-250, 280-296), `myagent/subagent/pool.py` (lines 248-263)
- **Description**: The `JsonLineFormatter.format()` reads `getattr(record, "extra_fields", None)` to extract category-specific fields. However, nearly all log call sites use Python's standard `extra={"category": "llm", "event": "request", ...}`, which Python's `logging` module expands as individual record attributes (`record.category`, `record.event`), NOT as a nested `record.extra_fields` dict. The formatter never picks up these fields because `record.extra_fields` is never set. Only the startup log in `LogManager.setup()` uses `extra={"extra_fields": {...}}`. The result is that all log records lack `category`, `event`, `model`, `latency_ms`, `tool_name`, and every other spec-required field in the JSON output.
- **Expected**: Either (a) all log call sites use `extra={"extra_fields": {"category": "llm", ...}}` consistently, or (b) the formatter reads common known fields directly from the record (`getattr(record, "category", None)`, etc.) and merges all unrecognized `__dict__` keys into the output. The design spec §十一 clearly defines each category's fields (e.g., `llm`: model, thinking_mode, messages_count, estimated_tokens, latency_ms, completion_tokens, prompt_tokens, retry_count, stream).
- **Actual**: Most log calls use `extra={"category": "agent", ...}` which sets `record.category` directly. The formatter only processes `record.extra_fields` (if present), so category fields are silently dropped from JSON output.

## Gap 3: [high] Compression summaries never persisted — `CompressionEngine._session_dir` never set

- **Category**: missing
- **Section**: 三、上下文分层结构 (持久化策略)
- **Files**: `myagent/context/compression.py` (line 31, lines 177-193), `myagent/agent/engine.py` (line 114), `myagent/cli/main.py` (line 114)
- **Description**: The compression engine's `_layer3_summarize()` calls `_persist_summary()`, which writes to `self._session_dir / "summaries" / "compact-NNN.md"`. However, `_session_dir` is initialized as `None` (line 31) and is never assigned by the engine or the CLI startup code. There is no public setter or attribute assignment to set `compression._session_dir` anywhere in the codebase. As a result, summaries are silently skipped (`if self._session_dir:` evaluates to `False`) and never written to disk.
- **Expected**: The engine (or session manager) should set `compression._session_dir` to the active session's directory path (`~/.myagent/sessions/<project>/<hash>/<session-id>/`) when the session starts. Alternatively, make `_session_dir` a public property with a setter.
- **Actual**: `_session_dir` stays `None` forever; all summary persistence is dead code.

## Gap 4: [high] No per-turn message persistence during agent execution

- **Category**: missing
- **Section**: 三、上下文分层结构 (持久化策略), 十、会话系统
- **Files**: `myagent/agent/engine.py` (lines 125-451), `myagent/cli/repl.py` (lines 253-281), `myagent/context/persistence.py` (lines 86-92)
- **Description**: The design spec mandates "所有记录必须持久化" (all records must be persisted). The `SessionStore.save_turn()` and `_write_transcripts()` methods exist but are never called during live agent execution. The engine builds its own local `messages` list from `session.get_recent_messages()` at the start of `_react_loop()`, then accumulates messages in that local list without ever writing them back to the session object. `_write_transcripts()` is only called at session creation and `save_turn()` is never invoked. If the process crashes mid-conversation, all in-flight messages are lost. Additionally, the `/history` slash command reads from `session._messages`, which will always be empty during a live session (only populated if resumed from disk).
- **Expected**: After each user-model exchange (or at minimum after each completed ReAct iteration), the engine should call `session.add_message()` and `session_store.save_turn()` to persist messages. The design spec's directory structure shows `transcript.json` and `transcript.md` per session that should contain the full conversation.
- **Actual**: Messages are only persisted at session creation and shutdown. All intermediate conversation data lives solely in memory.

## Gap 7: [medium] No mechanism for config runtime overrides from natural language conversation

- **Category**: missing
- **Section**: 九、配置系统 (7-level priority, runtime overrides at level 2)
- **Files**: `myagent/config/loader.py` (line 270-277), `myagent/agent/engine.py`
- **Description**: The design spec defines runtime overrides ("对话中自然语言调整") as priority level 2 in the 7-level config hierarchy. `ConfigLoader.apply_runtime_override()` exists and can update a config key-value pair. However, this method is never called from the agent engine or any other runtime component. When a user says "以后用 Think Max 模式" or "把并发数改成 5", there is no code path that detects the config change intent in the model's response and calls `apply_runtime_override()`. Only permission changes get runtime handling via `PermissionController.apply_runtime_rule()`. Config changes like thinking mode, concurrency limits, tool result char limits, etc., have no natural-language adjustment pathway.
- **Expected**: The engine should detect config-change intents in the model's response (or the model should be prompted to output a config-change tool call) and route them to `config.apply_runtime_override()`. Alternatively, provide a built-in tool `config_set` that the model can call.
- **Actual**: Only permission rules can be changed mid-conversation. All other config sections require CLI args or file edits.

## Gap 8: [medium] MemoryWriteTool bypasses MemoryStore — no dedup, no session log

- **Category**: incomplete
- **Section**: 六、记忆系统 + 梦境机制 (记忆生命周期)
- **Files**: `myagent/tools/builtin/memory_tools.py` (lines 26-56), `myagent/memory/store.py` (lines 58-97)
- **Description**: The `MemoryWriteTool.execute()` writes memory files directly to the filesystem using `path.write_text()`, with its own static `_update_index()` method. It does not use `MemoryStore.write()`, which provides critical functionality: (a) dedup by frontmatter name — checking for existing files with the same `name` in frontmatter and updating them instead of creating duplicates; (b) session memory logging — tracking created/updated/deleted memories in `_session_log` for the end-of-session summary. As a result, the agent can create duplicate memory files with different filenames but the same `name` in frontmatter, and the session-end memory change summary will never show memories created via the tool.
- **Expected**: `MemoryWriteTool.execute()` should delegate to `context.memory_store` (available via `ToolContext`) rather than writing files directly. The tool should receive a reference to the `MemoryStore` through the tool context.
- **Actual**: Tool writes files directly, bypassing `MemoryStore.write()` dedup and session logging.

## Gap 9: [medium] Log rotation uses date-prefixed filenames instead of TimedRotatingFileHandler

- **Category**: incomplete
- **Section**: 十一、日志系统 (实现概览)
- **Files**: `myagent/logging/logger.py` (lines 87-98, 191-217)
- **Description**: The design spec explicitly states: "日志文件通过 `logging.handlers.TimedRotatingFileHandler` 按天轮转 + `RotatingFileHandler` 按大小轮转组合". The implementation uses only `RotatingFileHandler` with date-prefixed filenames (e.g., `myagent-2026-07-03.jsonl`). While this achieves daily file separation (because the date is embedded in the filename and changes at midnight), it does NOT use Python's `TimedRotatingFileHandler` for time-based rotation. A long-running session that spans midnight would continue writing to the old date's file because the filename is computed once at startup.
- **Expected**: Use `TimedRotatingFileHandler` (with `when='midnight'`) to automatically rotate at midnight, combined with `RotatingFileHandler` for size-based rotation. Or at minimum recompute the date-based filename periodically.
- **Actual**: Single `RotatingFileHandler` with a statically computed date prefix.

## Gap 10: [medium] No mid-execution user interrupt capability in REPL

- **Category**: missing
- **Section**: 二、核心 Agent 循环 (用户交互)
- **Files**: `myagent/cli/repl.py` (lines 215-306), `myagent/agent/engine.py`
- **Description**: The design spec describes a natural-language interaction model where users can say "别改了，停下来" to stop the agent mid-operation. However, the REPL's `process_input()` is an `async for` loop that awaits the entire agent execution before accepting new input. There is no mechanism (e.g., asyncio task cancellation, signal handling, or a concurrent input reader) to interrupt a running agent. The user must wait for the current ReAct loop iteration to complete before their next input is processed.
- **Expected**: The REPL should run the agent engine in a background asyncio task and concurrently listen for user input. When the user types something during execution, the REPL should cancel the agent task and feed the interruption as a new input. Ctrl+C should trigger an interrupt signal.
- **Actual**: User input is fully blocked during agent execution. Ctrl+C in the REPL only resets the input buffer, not interrupts the running agent.

## Gap 12: [low] Task list is global (module-level), not session-scoped

- **Category**: incomplete
- **Section**: 四、工具系统 (会话工具)
- **Files**: `myagent/tools/builtin/session_tools.py` (lines 132-144)
- **Description**: The `_current_task_list` is a module-level global variable. `get_task_list()` returns this global, and `reset_task_list()` replaces it. This means all sessions within the same process share one task list. If a user runs `--resume` to switch sessions without restarting the process, tasks from the previous session leak into the new one.
- **Expected**: The task list should be stored on the session object or injected via `ToolContext`, so each session has its own independent task list. The `reset_task_list()` function exists but is never called from the session lifecycle code.
- **Actual**: Global mutable state shared across process lifetime.

## Notes

All 51 Python files under `myagent/` were read and compared against every section (一 through 十一) of the design spec. The categories `missing`, `incomplete`, and `deviation` were assigned based on whether a feature is absent, partially present, or implemented contrary to spec.

The four-layer architecture (CLI, Application, Service, Infrastructure) is structurally correct and all major components are wired. The most severe issues are in the dream engine's broken write calls (Gap 1, will crash at runtime), the logging formatter's field extraction mismatch (Gap 2, silently drops all spec-mandated category fields), and the missing per-turn persistence (Gap 4, data loss on crash). Compression summary persistence (Gap 3) is dead code until `_session_dir` is properly wired.
