## Blocking / Important Gaps

### G23-01
- **Severity**: Important
- **Design Requirement**: AGENTS requires every `except` block to log `category="error"` with `exception_type`, `traceback`, `context`, and `component` (`AGENTS.md:103`). The design spec defines `error` logs for any exception with those same fields (`docs/superpowers/specs/2026-07-02-myagentcli-design.md:985`).
- **Actual Implementation**: Round 22 hotspots were improved, but global compliance is still incomplete. A broad source scan returned 204 lines containing `except`; many real handlers still silently swallow exceptions or log as `debug`, `warning`, `system`, `subagent`, or `memory` without `exc_info=True`, `component`, and `context`.
- **Evidence**:
  - Session persistence swallows or falls back without error-category logs: duration parse fallback (`myagent/context/persistence.py:154`), tool-call markdown export `pass` (`myagent/context/persistence.py:273`), JSON export `pass` (`myagent/context/persistence.py:306`), transcript read fallback (`myagent/context/persistence.py:338`), duration fallback (`myagent/context/persistence.py:365`).
  - Context compression still has unlogged or non-error exception paths: Layer 3 failure accounting (`myagent/context/compression.py:122`), placeholder fallback (`myagent/context/compression.py:283`), best-effort persistence `pass` (`myagent/context/compression.py:305`, `myagent/context/compression.py:410`), and Layer 2 LLM summarization logged as debug/category agent rather than error (`myagent/context/compression.py:378`, `myagent/context/compression.py:383`, `myagent/context/compression.py:384`).
  - LLM provider has callback/prompt logging/token-count exceptions that are swallowed or lack required metadata (`myagent/llm/provider.py:262`, `myagent/llm/provider.py:373`, `myagent/llm/provider.py:410`, `myagent/llm/provider.py:515`, `myagent/llm/provider.py:698`, `myagent/llm/provider.py:748`). Its `except LLMError` fallback logs use `category="error"` and `component` but no traceback/context (`myagent/llm/provider.py:214`, `myagent/llm/provider.py:218`, `myagent/llm/provider.py:227`), and unknown litellm exceptions are wrapped without local logging (`myagent/llm/provider.py:417`).
  - CLI/MCP startup paths still log plain `_log.error` or `pass`: MCP startup/list failures (`myagent/cli/main.py:441`, `myagent/cli/main.py:470`, `myagent/cli/main.py:505`) and cleanup failure (`myagent/cli/main.py:509`) lack structured error metadata.
  - Sub-agent lifecycle has silent or non-error exception paths: transcript persistence (`myagent/subagent/pool.py:134`), resume scan (`myagent/subagent/pool.py:235`), status callback debug log (`myagent/subagent/pool.py:346`), shutdown/result-consumed callback `pass` (`myagent/subagent/pool.py:364`, `myagent/subagent/pool.py:381`), progress/retry callbacks (`myagent/subagent/worker.py:188`, `myagent/subagent/worker.py:383`), and worktree failures logged as category `subagent` without traceback (`myagent/subagent/worker.py:495`, `myagent/subagent/worker.py:515`).
  - Memory recall/store fallback paths also miss required error metadata (`myagent/memory/recall.py:52`, `myagent/memory/recall.py:115`, `myagent/memory/store.py:219`, `myagent/memory/store.py:269`).
- **Impact**: Operators still cannot reliably query every exception via `category="error"`, nor consistently recover traceback/context/component from JSONL logs. This violates both repository convention and the design logging contract even though user-facing fallback behavior may continue.
- **Suggested Fix**: Do a true repository-wide exception logging sweep, not only the round-22 files. Use a small helper or consistent pattern such as `logger.exception(..., extra={"category": "error", "component": "...", "context": "..."})`. For deliberately ignored expected control flow, either avoid catching exceptions or catch narrow types and still emit structured debug/error evidence according to the project convention. Add a static or behavioral regression test that fails for broad `except Exception: pass` and representative non-error exception logs.
