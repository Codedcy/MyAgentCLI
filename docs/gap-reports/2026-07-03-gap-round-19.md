## Gap 1: [high] LLM retry strategy deviates from spec
- **Category**: deviation
- **Section**: §二 核心Agent循环 — LLM 错误处理与重试
- **Files**: `myagent/llm/provider.py`
- **Description**: The design spec explicitly states "使用 LiteLLM 内置 retry 机制" for LLM retries. However, the code disables LiteLLM's built-in retry (`litellm.num_retries = 0`) and implements its own explicit exponential backoff loop in `_complete_with_model()`.
- **Expected**: The spec says retries should use LiteLLM's internal retry mechanism, with exponential backoff at 2s base delay, 30s max, max 3 retries. The spec expects LiteLLM's `num_retries` configuration to drive this behavior transparently.
- **Actual**: The code comments explain LiteLLM's internal tenacity-based retry uses hardcoded wait parameters (min=4s, max=10s) that don't match the spec. The code disables LiteLLM's retry entirely and implements explicit retry with the correct parameters (2s-30s). While functionally correct, the implementation layer differs from the specified approach — the retry is not "对上层透明" via LiteLLM's built-in mechanism as documented.

## Gap 2: [medium] Stream interruption doesn't prompt user for decision
- **Category**: incomplete
- **Section**: §二 核心Agent循环 — LLM 错误处理与重试 (流中断)
- **Files**: `myagent/agent/engine.py`, `myagent/cli/repl.py`
- **Description**: When a streaming LLM response is interrupted mid-way, the spec requires: "保留已接收内容 + 注入截断提示，允许用户决定是否继续". The engine correctly preserves partial content (gap-06) and emits an `IntentSignal(intent="continue")`, but the REPL layer does not provide an explicit interactive prompt asking the user whether to continue. The user only sees the truncated output in the console.
- **Expected**: After a stream interruption, the user should see an explicit prompt like "Stream interrupted. Continue? [Y/n]" so they can decide how to proceed.
- **Actual**: The engine yields `IntentSignal(intent="continue")` and returns. The REPL's `process_input` method finishes the `_run_engine` task and returns to waiting for the next user input. The user is not explicitly asked whether to continue, retry, or abandon.

## Gap 5: [high] SSE transport connect() method is broken/incomplete
- **Category**: incomplete
- **Section**: §四 工具系统 — MCP 集成 (SSE 传输)
- **Files**: `myagent/tools/mcp/client.py`
- **Description**: The `SSETransport.connect()` method creates three separate `httpx.AsyncClient` instances in sequence, performs `send`+`aclose`+`stream`+`aclose` on the first two, and then creates a third client for the actual streaming connection. The intermediate client creations are dead code — the responses are immediately closed without use. Additionally, the SSE stream retrieval (`self._sse_response.aiter_raw()`) may not correctly handle SSE event framing, and the background `_sse_reader` task has a fragile byte-level SSE parsing approach.
- **Expected**: The SSE transport should create a single `httpx.AsyncClient`, open one SSE stream via a streaming GET request, and use a robust SSE parser (e.g., `httpx_sse` or a well-tested state-machine parser) to extract events from the stream.
- **Actual**: Lines 192-241 show three separate client instances being created, two of which are immediately torn down. The `_sse_reader` (lines 309-355) manually parses SSE events from raw bytes with regex, which is fragile and doesn't handle all SSE edge cases (e.g., multi-line data fields, comments starting with `:`).

## Gap 6: [low] Tool result summarization passes full untruncated content to sub-agent
- **Category**: incomplete
- **Section**: §四 工具系统 — 工具调用流程
- **Files**: `myagent/agent/engine.py`
- **Description**: The spec describes tool result summarization as: "result > 5000 chars → spawn summarizer → summary + references". The implementation in `_summarize_via_subagent()` passes the FULL tool result (possibly extremely large, e.g., 500K+ chars) directly in the sub-agent prompt without any size limit. The code comment says "the sub-agent has a 1M context window" but there is no guard for results exceeding even 1M characters, and no mention of the file-reference fallback for very large results.
- **Expected**: The summarizer sub-agent should receive the full result for accurate summarization, but with a reasonable ceiling (e.g., 200K chars). Results exceeding that ceiling should be passed via file reference so the sub-agent can read them using the `read` tool.
- **Actual**: The full `result.output` is passed into the prompt string without truncation. The code comment mentions the file-reference approach for "extremely large results (>100K chars)" but this guard is not implemented in the prompt construction.

## Gap 7: [low] Context compression triggers hard truncation (L4) without going through compact() pipeline
- **Category**: incomplete
- **Section**: §三 上下文分层 — 压缩策略（四层渐进）
- **Files**: `myagent/agent/engine.py`
- **Description**: The spec's compression flowchart shows Layer 4 hard truncation as part of the 4-layer progressive sequence. In the engine's code (lines 302-331), the 90% hard truncation is implemented as a separate block that calls `self.compression._layer4_truncate()` directly, bypassing the `compression.compact()` pipeline. This means Layer 4 truncation doesn't benefit from the other layers (L1 cleanup, L2 summarization) that could potentially reduce context enough to avoid hard truncation.
- **Expected**: The 90% trigger should invoke `compression.compact()` which applies all applicable layers including Layer 4. The compact() method already handles Layer 4 as the final safety layer (lines 170-175 of compression.py).
- **Actual**: The engine duplicates the L4 truncation logic outside the compression pipeline. While this still achieves the goal of keeping context under 90%, it may result in unnecessarily aggressive truncation because it skips the gentler L1-L3 layers.

## Gap 8: [medium] Sub-agent pool spawn() wraps original but doesn't respect semaphore ordering for background task names
- **Category**: incomplete
- **Section**: §八 子Agent池 — 池管理
- **Files**: `myagent/cli/main.py`
- **Description**: In `main.py` (lines 268-291), the status bar integration monkey-patches `subagent_pool.spawn` with a wrapper that captures task names and fires status callbacks. However, the wrapper calls `_original_spawn` (which starts the background task) BEFORE storing the task name in `_task_names`. If the background task completes very quickly (before the wrapper stores the name), the status callback may fail to find the task name.
- **Expected**: The task name should be stored in `_task_names` before `_original_spawn` is called, or the callback should gracefully handle missing task names.
- **Actual**: The ordering is: `handle = await _original_spawn(...)` → `_task_names[handle.id] = name` → status callback. For background-spawned sub-agents, `_original_spawn` returns immediately after creating the background task, so the timing issue is unlikely but the ordering still places storage after spawn, which is logically inverted.

## Gap 9: [low] Prompt-based tool call "name" field omitted in transcript persistence for tool-result messages
- **Category**: incomplete
- **Section**: §三 上下文分层 — 持久化策略, §十 会话系统
- **Files**: `myagent/agent/engine.py`, `myagent/context/persistence.py`
- **Description**: The engine's `_persist_turn` method persists tool result messages to session transcripts. When converting dict-format messages to `Message` objects, it correctly extracts `tool_call_id` and `name` fields. However, the `Session` class's `add_message` increments `turn_count` for every message (including tool results), which means each tool call counts as a separate "turn." The spec's turn counting should likely be based on user-assistant exchange pairs, not individual messages.
- **Expected**: Turn counting should reflect conversation rounds (user→assistant pairs), not individual internal messages (tool calls, tool results).
- **Actual**: `Session.add_message` increments `turn_count += 1` unconditionally for every message, including tool results. This inflates the turn count and could cause the dream engine's trigger threshold (50 rounds) to fire earlier than intended.

## Gap 10: [medium] Missing MCP server resource/prompt list calls during startup for non-tool MCP capabilities
- **Category**: missing
- **Section**: §四 工具系统 — MCP 集成
- **Files**: `myagent/cli/main.py`, `myagent/tools/mcp/client.py`
- **Description**: The spec's MCP integration sequence diagram explicitly shows: after `tools/list`, the startup calls `resources/list` and `prompts/list`. The code in `_start_single_mcp_server()` does call `client.list_resources()` and `client.list_prompts()`, but wraps these in bare `except Exception: pass` blocks. If an MCP server doesn't implement these optional methods and returns a JSON-RPC error (not an exception), the error response is silently ignored but the results are not stored. More importantly, errors from `tools/list` (a required MCP method) are caught by the same broad exception handler, which could mask genuine tool discovery failures.
- **Expected**: The startup sequence should distinguish between "method not supported" (acceptable for optional resource/prompt endpoints) and genuine failures (transport errors, malformed responses). The `tools/list` call should not be silenced — if it fails, the server should be treated as broken.
- **Actual**: All three discovery calls (tools/list, resources/list, prompts/list) are wrapped in a broad `except Exception: pass` pattern that silences all failures equally, making it impossible to distinguish between "no prompts" and "server crashed during prompt listing."
