# Gap Report — Round 14

## Gap 1: [medium] Skill resources — reference/script paths exposed as filenames only, not full paths; templates and assets not listed
- **Category**: incomplete
- **Section**: §七 技能系统 — 脚本和 references 的处理方式
- **Files**: myagent/context/builder.py, myagent/skills/loader.py
- **Description**: The `_format_skill_content` method in ContextBuilder lists skill references and scripts using `str(r.name)` (filename only) instead of `str(r)` (absolute path). The agent sees only filenames like `security-checklist.md` without knowing which directory to read from. Additionally, `templates/` and `assets/` resource directories are not listed at all — only `references/` and `scripts/` are enumerated.
- **Expected**: Per spec §七: "invoke 技能时，references 文件列表作为'可用参考'注入 system prompt，Agent 需要时自行 tool_call: read('{skill_dir}/references/xxx.md') 按需读取". The full (or relative-to-skill-dir) path should be provided for references, scripts, templates, and assets so the agent can actually read them.
- **Actual**: `_format_skill_content` yields: `"References: security-checklist.md"` and `"Scripts: lint.sh"` — no path context. Templates and assets are completely absent from the system prompt.

## Gap 2: [medium] Skill model missing base_dir attribute for path resolution
- **Category**: incomplete
- **Section**: §七 技能系统 — 技能目录结构
- **Files**: myagent/skills/loader.py (Skill dataclass), myagent/context/builder.py
- **Description**: The `Skill` dataclass stores `name`, `description`, `content`, and `resources` but does NOT store the skill's base directory (`base_dir`). Without this, even if resource Path objects contain absolute paths via SkillLoader, the agent has no programmatic way to locate the skill directory to read resources or execute scripts. The spec says "脚本路径相对于技能目录根" which implies the path relationship must be traceable.
- **Expected**: The Skill dataclass should include a `base_dir: Path` field set by SkillLoader.parse_skill_md, so the context builder can construct actionable paths for the agent.
- **Actual**: Skill has no `base_dir`; resources Path objects are set but not used with path context in the system prompt.

## Gap 3: [low] Bash tool has redundant permission checks causing double confirmation
- **Category**: deviation
- **Section**: §五 权限/沙箱系统 — 确认流程
- **Files**: myagent/tools/builtin/exec_tools.py (BashTool.execute), myagent/agent/engine.py (_execute_tool)
- **Description**: The engine's `_execute_tool` method already performs a full permission check (check → ALLOW/DENY/ASK → confirm) for every tool before calling `tool.execute()`. However, `BashTool.execute` independently runs its own permission check (`permissions.check` → `permissions.confirm`) unless `dangerouslyDisableSandbox=True`. This results in a double confirmation dialog for bash commands: the user is first prompted by the engine, then again by the BashTool itself. Both prompts are identical.
- **Expected**: Permission checks should happen in exactly one place — the engine (centralized). Tool implementations should trust the engine's pre-check and not re-prompt the user. The `dangerouslyDisableSandbox` flag should bypass the engine check (not the other way around).
- **Actual**: Engine checks → BashTool checks again → user sees two permission prompts for the same bash command.

## Gap 4: [low] Sub-agent pool ID counter resets on session resume, risking transcript directory conflicts
- **Category**: incomplete
- **Section**: §八 子 Agent 池与工作流编排 — 池管理, §三 上下文分层 — 持久化策略
- **Files**: myagent/subagent/pool.py (SubAgentPool.spawn), myagent/agent/main.py (async_main), myagent/context/persistence.py
- **Description**: When a session is resumed via `--resume`, the SubAgentPool is created fresh with `_counter = 0`. Sub-agents spawned in the resumed session receive IDs like `sub-001`, `sub-002`, etc. If the original session run already created sub-agents with these same IDs, the new sub-agents will overwrite the previous `subagents/sub-NNN/` transcript directories because `_persist_subagent_transcript` uses `mkdir(exist_ok=True)` and overwrites `transcript.json`/`transcript.md`. This loses the original sub-agent transcript history.
- **Expected**: The SubAgentPool should either (a) scan existing sub-agent directories on resume and set its counter above the highest existing ID, or (b) use a globally unique ID scheme (e.g., include timestamp) so resumed sessions never collide with previous transcript directories.
- **Actual**: Pool counter always starts at 0 regardless of existing sub-agent transcript directories in the resumed session.

## Gap 5: [low] Compression engine's minimum_messages guard bypassed in engine _react_loop for 75% trigger
- **Category**: incomplete
- **Section**: §三 上下文分层结构 — 压缩策略（防抖保护）
- **Files**: myagent/agent/engine.py (_react_loop), myagent/context/compression.py (compact method)
- **Description**: The spec's debounce rules state "最少 10 轮消息才触发压缩". The `CompressionEngine.compact` method correctly checks `minimum_messages` and returns early if the message count is below the threshold. However, the engine's `_react_loop` calls compaction at 75% usage regardless — and the compact method's minimum_messages guard only uses `self.config` which is the compression config dataclass. If the compression config is properly set, this works. But the engine also adds a system message at position 0 before counting messages, meaning the effective message count for the guard is off by one. This is a minor edge case.
- **Expected**: The message count check should be based on actual conversation messages (excluding the system prompt insertion).
- **Actual**: The system prompt is inserted at `messages[0]` in the engine before calling compact, inflating the message count by 1. This could cause compression to trigger one turn earlier than expected in edge cases.
