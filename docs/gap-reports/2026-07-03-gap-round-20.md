# Gap Report — Round 20

## Summary
- 缺失功能 (missing): 1
- 不完整实现 (incomplete): 5
- 偏离设计 (deviation): 3

## Gap 1: [low] Sub-agent worker lacks context compression
- **Category**: incomplete
- **Section**: §三 上下文分层结构
- **Files**: myagent/subagent/worker.py
- **Description**: The main agent engine has a full 4-layer progressive compression pipeline (L1 cleanup, L2 tool summarization, L3 conversation summary, L4 hard truncation) triggered at 75% usage with a 90% hard limit. However, SubAgentWorker._run_impl() has no context management whatsoever — it has MAX_ITERATIONS=30 but no compression checks, no usage estimation, and no truncation. The design spec §三 states that sub-agents inherit the same context window as the main agent ("跟随模型上限，自动继承主Agent"). If a sub-agent spawns with a large prompt and executes many tool calls, it could silently exhaust the 1M token context window with no mitigation.
- **Expected**: SubAgentWorker should include at minimum a basic context-size guard (e.g., message count limit or character-based truncation), ideally lightweight compression aligned with the main agent's pipeline.
- **Actual**: SubAgentWorker runs a plain while-iteration loop with no context usage monitoring or compression.

## Gap 2: [low] LLM retry backoff parameters may differ from spec
- **Category**: deviation
- **Section**: §二 核心 Agent 循环 — LLM 错误处理与重试
- **Files**: myagent/llm/provider.py
- **Description**: The design spec explicitly states retry parameters: "指数退避重试，最多 3 次，初始间隔 2s，上限 30s". However, the code delegates to LiteLLM's built-in retry mechanism (via `litellm.num_retries = MAX_RETRIES`), which internally uses tenacity with its own default backoff parameters. The spec also says "使用 LiteLLM 内置 retry 机制" in the same paragraph, creating an internal conflict. The net result is that the actual backoff timing is controlled by tenacity's defaults, not the explicitly specified 2s/30s values.
- **Expected**: Either the backoff should use the exact spec parameters (2s base, 30s max) via litellm configuration, or the spec's explicit timing requirements should be clarified as non-binding when litellm's built-in retry is active.
- **Actual**: litellm.num_retries is set to 3, but backoff interval and max delay are determined by tenacity's internal defaults, which may not match the spec's 2s/30s.

## Gap 3: [low] Skill resource paths are absolute, not relative
- **Category**: deviation
- **Section**: §七 技能系统 — 脚本和 references 的处理方式
- **Files**: myagent/skills/loader.py, myagent/context/builder.py
- **Description**: The spec §七 states "脚本路径相对于技能目录根". SkillLoader.enumerate_resources() collects absolute Path objects via `sorted(subdir.iterdir())`, and ContextBuilder._format_skill_content() formats these absolute paths into the system prompt. While functionally correct (absolute paths work everywhere), the spec explicitly calls for relative paths from the skill directory root. This means the model sees something like `/home/user/.myagent/skills/code-review/scripts/lint.sh` instead of `scripts/lint.sh`, which is longer and leaks filesystem layout into the context.
- **Expected**: Skill resource paths should be relative to the skill directory root, e.g. `scripts/lint.sh` instead of absolute paths.
- **Actual**: All resource paths are absolute Path objects from `iterdir()`.

## Gap 4: [low] Session command-line help text missing export usage
- **Category**: incomplete
- **Section**: §三 上下文分层结构 — 会话管理命令, §十 会话系统与项目感知
- **Files**: myagent/cli/main.py
- **Description**: The spec §三 documents `myagent --session <session-id> --export markdown` as a CLI usage pattern. While the argparse definitions in parse_args() do support `--session` and `--export`, the argparse help text and the overall CLI design do not surface this usage pattern clearly. There is no usage example or extended help explaining that `--session` and `--export` must be combined. A user reading `--help` sees separate flags with no indication they work together.
- **Expected**: argparse epilog or help text should document the combined usage: `myagent --session <id> --export markdown`.
- **Actual**: `--session` and `--export` are defined as independent flags with no cross-reference in help text.

## Gap 5: [low] No per-skill invocation metrics tracking
- **Category**: missing
- **Section**: §七 技能系统 — 技能调用实现机制
- **Files**: myagent/agent/engine.py, myagent/cli/commands.py
- **Description**: The spec describes skill invocation as: model emits `skill_invoke` virtual tool call, engine intercepts it, loads full SKILL.md into system prompt, and model follows skill instructions in the next ReAct iteration. The code implements this flow correctly (engine._react_loop intercepts `skill_invoke` and rebuilds context with active_skill). However, there is no tracking or metrics for skill invocations — no logging of which skill was invoked, how many times, or whether the skill completed successfully. This makes it impossible to analyze skill usage patterns or measure effectiveness.
- **Expected**: Skill invocations should be logged (category="skill", event="invoked") with skill name, and ideally with completion status.
- **Actual**: Skill invocations are handled silently with no dedicated logging category.

## Gap 6: [low] subagent.send_message "to=main" response format is underspecified
- **Category**: incomplete
- **Section**: §八 子 Agent 池与工作流编排
- **Files**: myagent/tools/builtin/agent_tools.py
- **Description**: The design spec §八 describes `send_message` as a tool for "向子 Agent 或主 Agent 发消息". The SendMessageTool.execute() handles `to="main"` for sub-agent-to-main-agent communication. However, when `pool.send_to_main()` enqueues the message, the sub-agent's own ID is retrieved from `params.get("from", "subagent")` — but the `from` field is not part of the send_message tool schema. The calling sub-agent must add a non-schema `from` parameter. This is a fragile implicit contract.
- **Expected**: Either the `from` field should be in the tool schema, or the sub-agent worker should automatically inject its own ID when invoking send_message.
- **Actual**: The `from` field is an undocumented param that sub-agents must discover and populate themselves.

## Gap 7: [low] PermissionController.confirm() bypass in non-TTY environments
- **Category**: deviation
- **Section**: §五 权限/沙箱系统 — 确认流程
- **Files**: myagent/permissions/controller.py
- **Description**: The spec §五 emphasizes that "权限确认不设超时——一直等待用户明确响应". However, PermissionController.confirm() checks `sys.stdin.isatty()` and auto-allows in non-interactive environments (tests, CI, piped stdin). While this is a practical safety measure for automated contexts, it directly contradicts the spec's intent of never auto-allowing without explicit consent.
- **Expected**: Non-TTY environments should either refuse to execute privilege-escalating operations, or at minimum log a prominent warning. The spec doesn't mention a non-TTY fallback.
- **Actual**: Non-TTY environments auto-allow all tools via `return True` with only a logger.warning.

## Gap 8: [medium] Dream engine hours-based trigger uses file mtime, not explicit timestamp
- **Category**: incomplete
- **Section**: §六 记忆系统 + 梦境机制
- **Files**: myagent/memory/dream.py
- **Description**: The spec §六 states dream trigger is: "Agent 启动时，距离上次梦境 > 6 小时 AND 累计对话轮数 > 50 轮". The DreamEngine.should_run() checks `last_run` from last_dream.json for the time check. However, it also writes `round_count: 0` after each dream (resetting the counter). If the agent crashes without updating last_dream.json, the `round_count` can be reset while `last_run` is stale. Additionally, the dream engine estimates total rounds by scanning all transcript files' turn_count fields, which counts ALL historical sessions, not just those since the last dream. This means if a user has 1000 rounds across all historical sessions, the dream triggers every session (as long as hours > 6), even if only 5 rounds happened since the last dream.
- **Expected**: The round counter should track rounds since the last dream completion, not total historical rounds. The spec says "累计对话轮数 > 50 轮" in context of triggering — this should mean 50 rounds accumulated SINCE the last dream, not 50 rounds total across all time.
- **Actual**: estimate_total_rounds() sums ALL session transcript turn_count values globally, making the 50-round threshold trivially satisfied for any user with more than a few historical sessions.

## Gap 9: [low] ContextBuilder memory cache key collision risk
- **Category**: incomplete
- **Section**: §六 记忆系统 + 梦境机制 — 记忆生命周期
- **Files**: myagent/context/builder.py
- **Description**: The ContextBuilder uses a session-scoped memory cache with `_cache_key = query[:100]` (first 100 chars of user input). If two different user inputs share the same first 100 characters but diverge in topic thereafter, the cache returns stale memories from the first query. This is unlikely in practice but technically a correctness gap. The drift detection (`_detect_topic_drift`) mitigates this partially by checking keyword overlap on the full input, but the cache key itself is still a simple prefix.
- **Expected**: The cache key should incorporate a content hash (e.g., hash of significant keywords) rather than a raw prefix to avoid false cache hits on long inputs with common prefixes.
- **Actual**: `_cache_key = query[:100]` uses raw string prefix as cache key.
