## Blocking / Important Gaps

### G27-01
- **Severity**: Blocking / Permission and Logging Contract
- **Design Requirement**: Every tool call flows through permission checking before `Tool.execute()` (`docs/superpowers/specs/2026-07-02-myagentcli-design.md:394`-`:402`), with write/exec/network tools requiring confirmation as configured (`:447`-`:450`). Every tool execution log must include `tool_name`, `params_summary`, `permission_result`, `duration_ms`, `result_size_chars`, and optional `error` (`:982`; also implementation plan `docs/superpowers/plans/2026-07-03-myagentcli-implementation.md:530`, `:557`-`:563`).
- **Actual Implementation**: Main-agent tool execution is guarded, but `SubAgentWorker` executes tool calls directly. It looks up a tool and calls `tool.execute(tc.params, ctx)` without `context.permissions.check()` / `confirm()`, and logs only `extra={"category": "tool"}` for successful child tool calls.
- **Evidence**:
  - Main path checks permissions in `myagent/agent/engine.py:871`-`:887`.
  - Sub-agent path calls tools directly in `myagent/subagent/worker.py:257`-`:275`.
  - Sub-agent success log lacks the required fields in `myagent/subagent/worker.py:282`-`:288`.
  - `ToolContext` carries a permissions controller (`myagent/tools/base.py:47`, `:56`), but the sub-agent worker does not use it before execution.
  - Focused probe: a sub-agent with inherited `permissions.check.return_value = DENY` still executed a fake `write` tool; output showed `tool_called True` and `permission_check_calls 0`.
- **Impact**: A child agent can execute write/exec/MCP tools without the per-call permission gate or the required audit metadata. This can bypass level 1/2/3 decisions after the parent only approved `spawn_subagent`, and it leaves sub-agent tool activity under-logged.
- **Suggested Fix**: Factor a shared tool-execution helper used by both `AgentEngine` and `SubAgentWorker`, or implement the same permission and structured tool-log contract in `SubAgentWorker`. Add regression tests for sub-agent permission denial, confirmation behavior, and tool-log field coverage.

### G27-02
- **Severity**: Important / MCP Configuration Contract
- **Design Requirement**: MCP startup reads user-level and project-level `mcp.json` and starts each configured server (`docs/superpowers/specs/2026-07-02-myagentcli-design.md:424`-`:433`). The implementation plan documents the config shape as top-level `mcpServers` (`docs/superpowers/plans/2026-07-03-myagentcli-implementation.md:444`-`:458`).
- **Actual Implementation**: `_startup_mcp_servers()` supports top-level `servers` or a raw server dictionary, but not the documented `mcpServers` key. Tests only cover `servers`.
- **Evidence**:
  - Parser uses `servers = data.get("servers", data)` in `myagent/cli/main.py:441`, then iterates the resulting dict at `:442`-`:444`.
  - `_start_single_mcp_server()` expects each server config to contain `command` and skips if missing (`myagent/cli/main.py:480`-`:483`).
  - Existing tests write `{"servers": ...}` only (`tests/cli/test_main.py:55`-`:57`, `:83`-`:85`, `:91`-`:93`).
  - Focused probe with documented `{"mcpServers": {"doc-server": {"command": "doc-cmd"}}}` showed `_startup_mcp_servers()` attempting to start a server named `mcpServers` with the nested dict rather than `doc-server`.
- **Impact**: Users following the documented MCP config will not get their MCP server started, so MCP tools/resources/prompts are absent despite valid-looking configuration.
- **Suggested Fix**: Accept `mcpServers` as the primary documented key, optionally keep `servers`/raw dict as compatibility aliases, and add tests for project/user merge and project override using `mcpServers`.

### G27-03
- **Severity**: Important / Dream Manual Trigger
- **Design Requirement**: Dream can be manually triggered by `/dream` or natural language review, and the dream flow scans unprocessed session transcripts before updating memories (`docs/superpowers/specs/2026-07-02-myagentcli-design.md:549`-`:568`).
- **Actual Implementation**: The `/dream` slash command calls `ctx.dream_engine.run()` without a `session_store`, so manual dream runs cannot scan persisted transcripts. Startup and periodic dream paths do pass a session store.
- **Evidence**:
  - `/dream` dispatch calls `await ctx.dream_engine.run()` in `myagent/cli/commands.py:123`-`:126`.
  - Dream transcript excerpt building returns “No session store available” when no store is passed (`myagent/memory/dream.py:385`-`:386`).
  - Startup background dream passes `session_store` (`myagent/cli/main.py:644`-`:649`), and periodic dream passes it too (`myagent/cli/repl.py:744`-`:748`).
- **Impact**: Manual `/dream` can produce a log while skipping the recent-session transcript analysis that the design says is part of the dream mechanism.
- **Suggested Fix**: Pass the active `SessionStore` from command context, for example via `ctx.engine.session_store` or `ctx.session_manager.session_store`, and add a slash-command test that verifies `DreamEngine.run(session_store=...)` is called.

### G27-04
- **Severity**: Important / Dream Sub-agent Tool Context
- **Design Requirement**: Dream updates memory files and `MEMORY.md` through the memory subsystem after transcript/memory analysis (`docs/superpowers/specs/2026-07-02-myagentcli-design.md:560`-`:568`).
- **Actual Implementation**: The LLM-driven Dream path spawns a sub-agent with `memory_write`, `read`, and `glob`, but does not pass a `ToolContext` containing `memory_store`, `project_dir`, config, or permissions. In CLI startup, the pool itself also has no default `tool_context`. Therefore Dream sub-agent `memory_write` can fall back to direct filesystem writes instead of `MemoryStore.write()`.
- **Evidence**:
  - Dream sub-agent spawn omits `tool_context` and `config` (`myagent/memory/dream.py:236`-`:241`).
  - CLI constructs `SubAgentPool` without a pool-level `tool_context` (`myagent/cli/main.py:122`-`:126`).
  - Worker fallback creates `ToolContext(..., permissions=None, config=self._config, subagent_pool=...)` without `memory_store` (`myagent/subagent/worker.py:263`-`:272`).
  - `MemoryWriteTool` uses `context.memory_store` only when present, otherwise writes directly to disk (`myagent/tools/builtin/memory_tools.py:42`-`:63`).
- **Impact**: Dream memory maintenance can bypass `MemoryStore` deduplication, `MEMORY.md` index updates, and session write accounting. This also compounds G27-01 because Dream sub-agent tools are not permission-gated or fully logged.
- **Suggested Fix**: Build and pass a Dream-specific `ToolContext` containing the real `memory_store`, project/user memory root, config, tool registry, permissions policy appropriate for silent dream work, and sub-agent pool. Add tests that a Dream sub-agent memory write uses `MemoryStore.write()` and updates operation counts/index state.
