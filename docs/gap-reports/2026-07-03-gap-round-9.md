## Gap 1: [high] session.save_transcripts config never honored -- transcripts always written unconditionally
- **Category**: incomplete
- **Section**: 九 配置系统, 十 会话系统
- **Files**: myagent/config/schema.py (line 78), myagent/context/persistence.py (lines 82, 92, 406-456), myagent/agent/engine.py (line 959)
- **Description**: SessionConfig defines `save_transcripts: bool = True` but the value is never checked anywhere in the persistence or engine code. `_write_transcripts()` is called unconditionally in `SessionStore.create_session()`, `SessionStore.save_turn()`, and `AgentEngine._persist_turn()`. The config key exists in `config_set` valid_keys but toggling it has no observable effect.
- **Expected**: When `session.save_transcripts = false`, transcript files (transcript.json, transcript.md) should NOT be written to disk. Only in-memory session state should be maintained.
- **Actual**: `_write_transcripts()` has no guard checking the config value. All transcript writes proceed unconditionally regardless of the setting.

## Gap 2: [high] session.transcript_format config never honored -- both formats always written
- **Category**: incomplete
- **Section**: 九 配置系统, 十 会话系统
- **Files**: myagent/config/schema.py (lines 79-81), myagent/context/persistence.py (lines 406-456)
- **Description**: SessionConfig defines `transcript_format: list[str] = ["json", "markdown"]` allowing users to choose which formats to write. The `_write_transcripts()` method always writes BOTH `transcript.json` AND `transcript.md` with no check against this config value. If the user sets `transcript_format: [json]`, Markdown transcripts should not be written.
- **Expected**: `_write_transcripts()` should check `config.session.transcript_format` and only write the formats that are listed (e.g. `"json" in transcript_format` before writing JSON).
- **Actual**: Both JSON and Markdown are always written with no format check.

## Gap 3: [high] Dream engine runs inline with rule-based analysis instead of spawning LLM-driven sub-agent
- **Category**: deviation
- **Section**: 六 记忆系统, 梦境机制
- **Files**: myagent/memory/dream.py (entire file), myagent/cli/main.py (lines 128-141, 532-544)
- **Description**: The design spec flowchart (§六 line 555-557) explicitly shows the dream cycle as spawning a background sub-agent: `DREAM[spawn 后台子Agent] --> SCAN[扫描未处理的 session transcripts] --> ANALYZE[分析]`. This implies an LLM-driven analysis where an agent reads transcripts and makes intelligent decisions about memory creation, merging, and deletion. The current DreamEngine runs entirely inline via `asyncio.create_task(_run_dream_background(...))` which calls `dream_engine.run()` synchronously. The analysis is purely rule-based: keyword matching for contradiction detection, file mtime for staleness, character counting for empty detection. The code even acknowledges this limitation at line 375: "Full semantic contradiction detection would require LLM analysis."
- **Expected**: Dream cycle should spawn a background sub-agent (with its own context and LLM) that scans recent session transcripts, analyzes them with LLM reasoning, and produces memory changes. The sub-agent approach allows deeper analysis (detecting semantic contradictions, understanding correction patterns, extracting conventions) that keyword matching cannot achieve.
- **Actual**: `DreamEngine.run()` performs direct file I/O and string matching without any LLM involvement. The dream runs as a regular asyncio task, not as a sub-agent with ReAct loop.

## Gap 4: [medium] ui.syntax_highlight config has no implementation
- **Category**: incomplete
- **Section**: 九 配置系统
- **Files**: myagent/config/schema.py (line 73), myagent/cli/repl.py (lines 167-204)
- **Description**: UIConfig defines `syntax_highlight: bool = True` but there is no code anywhere that implements syntax highlighting in the REPL. The `PromptSession` in `REPLEngine.run()` does not configure a `lexer` or syntax highlighter. The config key is a dead option -- setting it to True or False has no effect on the user experience.
- **Expected**: When `syntax_highlight = true`, the prompt_toolkit `PromptSession` should be configured with a `PygmentsLexer` (or similar) for the input language, or at minimum code blocks in the output should be syntax-highlighted via Rich's `Syntax` renderable.
- **Actual**: No syntax highlighting code exists. The config option is accepted and validated but never consumed.

## Gap 5: [medium] ui.streaming config not honored -- streaming is always on
- **Category**: incomplete
- **Section**: 九 配置系统
- **Files**: myagent/config/schema.py (line 72), myagent/agent/engine.py (lines 313-318), myagent/llm/provider.py (line 227)
- **Description**: UIConfig defines `streaming: bool = True` but the value is never checked. The LLM provider always sets `stream: True` in the LiteLLM kwargs, and the engine always processes the response as a stream. If a user sets `ui.streaming = false`, the system should fall back to non-streaming completion (collect full response then display).
- **Expected**: When `streaming = false`, the LLM provider should set `stream: False` and collect the full response before returning a single text event instead of streaming deltas.
- **Actual**: `stream: True` is hardcoded. The config option has no effect.

## Gap 6: [medium] MCP resources/prompts exposed in system prompt but no tool exists to read or invoke them
- **Category**: incomplete
- **Section**: 四 工具系统, MCP 集成
- **Files**: myagent/context/builder.py (lines 315-361), myagent/tools/mcp/client.py (lines 221-233), myagent/tools/registry.py (lines 36-37)
- **Description**: MCP resources and prompts are discovered at startup (`client.list_resources()`, `client.list_prompts()`) and exposed in the system prompt via `ContextBuilder._format_mcp_refs()`. However, there is no built-in tool for the model to actually CALL these. The model sees resource names in the system prompt but has no mechanism to read a resource's content (`mcp_read_resource`) or invoke a prompt template (`mcp_get_prompt`). The `MCPClient` has no `read_resource()` or `get_prompt()` methods.
- **Expected**: Two additional built-in MCP integration tools should exist: `mcp_read_resource` (takes a resource URI, calls the MCP server's `resources/read` method, returns content) and `mcp_get_prompt` (takes a prompt name and optional arguments, calls `prompts/get`, returns the rendered prompt). Alternatively, the existing MCP tool adapters should be extended to cover resources and prompts.
- **Actual**: MCP resources and prompts are informational only -- the model can read about their existence in the system prompt but cannot interact with them.

## Gap 7: [low] No /compact slash command for manual context compression
- **Category**: missing
- **Section**: 三 上下文分层, 压缩策略
- **Files**: myagent/cli/commands.py (lines 35-43), myagent/context/compression.py
- **Description**: The design spec (§三 line 270) states "50%: 推荐手动 compact" as a user-visible recommendation level. While the engine does emit a notification at 50% usage suggesting `/clear`, there is no `/compact` slash command that triggers the compression engine. `/clear` removes all in-memory messages, which is a destructive operation; compression (L1-L3) would be a non-destructive alternative that summarizes rather than discards.
- **Expected**: A `/compact` slash command should be available that triggers `CompressionEngine.compact()` on the current conversation messages, producing the same Layer 1-3 compression that auto-compact performs at 75%. This gives users an explicit, non-destructive way to free context space.
- **Actual**: Only `/clear` exists, which wipes all in-memory messages. No way to manually trigger compression.

## Gap 8: [low] config_set tool valid_keys list is hardcoded and incomplete
- **Category**: incomplete
- **Section**: 九 配置系统
- **Files**: myagent/tools/builtin/config_tools.py (lines 56-76)
- **Description**: The `valid_keys` set in `ConfigSetTool.execute()` is manually maintained and not derived from the `AppConfig` dataclass schema. Adding a new config field requires updating this list in addition to the schema. Additionally, several valid config paths are missing: `session.sessions_dir`, `model.provider`, `model.model`, `ui.status_bar_items`. This means these config values cannot be changed at runtime via `config_set`.
- **Expected**: The valid_keys should either be derived from the AppConfig schema programmatically, or include all documented config paths from the YAML config spec (§九). At minimum, `session.sessions_dir`, `model.provider`, `model.model`, and `ui.status_bar_items` should be added.
- **Actual**: 33 keys are listed manually. Four valid config paths from the spec are missing. The list must be manually kept in sync with the schema.

## Gap 9: [low] memory_write dedup only checks frontmatter name, not semantic content
- **Category**: incomplete
- **Section**: 六 记忆系统, 记忆生命周期
- **Files**: myagent/memory/store.py (lines 58-97, especially 155-166)
- **Description**: The design spec (§六 line 541) says memory_write should "检查是否有覆盖同一事实的旧文件 -- 更新而非重复创建". The current `MemoryStore.write()` dedup logic only checks if another memory file has the same frontmatter `name` field. It does not check for semantic overlap in the body content. Two memory files with different names but covering the same fact (e.g., "coding-style" and "python-conventions" both describing the same snake_case rule) would not be detected as duplicates.
- **Expected**: In addition to name-based dedup, the write path should perform content similarity checks (e.g., comparing descriptions or using the embedding model from recall.py) to detect semantically overlapping facts and update rather than create duplicates.
- **Actual**: Dedup is name-only via `_find_by_name()`. No content-level similarity checks are performed.

## Gap 10: [low] Dream engine contradiction detection is keyword-based only, not semantic
- **Category**: deviation
- **Section**: 六 记忆系统, 梦境机制
- **Files**: myagent/memory/dream.py (lines 371-420)
- **Description**: The spec flowchart (§六) shows the dream cycle performing "矛盾记忆? -- 合并去重" using sub-agent analysis. The current `_detect_contradictions()` method uses a hardcoded list of word pairs (`("use", "avoid"), ("always", "never"), ("prefer", "avoid"), ("支持", "不支持"), ("推荐", "避免")`) to detect contradictions. This is a fragile heuristic that misses semantically contradictory statements (e.g., "Use type hints" vs "Type hints are optional for internal code"). The method itself notes at line 375: "Full semantic contradiction detection would require LLM analysis."
- **Expected**: Contradiction detection should leverage LLM analysis (either directly or via sub-agent) to identify semantically contradictory statements across memory files, not just exact keyword occurrences.
- **Actual**: Purely heuristic keyword matching with a list of 5 predefined contradiction pairs. Many real-world contradictions will be missed.
