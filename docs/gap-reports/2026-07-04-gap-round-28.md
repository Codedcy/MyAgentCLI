## Blocking / Important Gaps

### G28-01
- Severity: Important / Logging Contract
- Design Requirement: Tool category logs must carry `tool_name`, `params_summary` truncated to 200 characters, `permission_result`, `duration_ms`, `result_size_chars`, and optional `error` (`docs/superpowers/specs/2026-07-02-myagentcli-design.md:982`; `AGENTS.md:99`). This round also explicitly asked whether successful and refused sub-agent tool logs satisfy that contract.
- Actual Implementation: Successful sub-agent tool logs include the required fields, but denied/refused sub-agent tool logs only include `category`, `tool_name`, `params_summary`, and `permission_result`.
- Evidence:
  - DENY log extras are missing `duration_ms`, `result_size_chars`, and `error` (`myagent/subagent/worker.py:308`-`:317`).
  - ASK rejected-by-user log extras are missing the same fields (`myagent/subagent/worker.py:339`-`:348`).
  - Focused probe result: `tool_executed False`, `permission_result denied`, `has_duration_ms False`, `has_result_size_chars False`.
- Impact: Refused sub-agent tool calls are permission-safe, but the audit trail is incomplete and inconsistent with successful tool execution logs. Operators cannot uniformly query tool latency/size/error fields across allowed and refused sub-agent tool records.
- Suggested Fix: Start timing before permission checks, log refusal records with `duration_ms`, `result_size_chars=0`, and the refusal error text, and add regression tests for DENY and rejected ASK log field completeness.

### G28-02
- Severity: Important / Dream Silent Background Contract
- Design Requirement: Dream runs as a background memory-consolidation mechanism and must not ask for confirmation (`docs/superpowers/specs/2026-07-02-myagentcli-design.md:545`-`:577`; `AGENTS.md:65`). It should update memory files and `MEMORY.md` as part of that background flow (`docs/superpowers/specs/2026-07-02-myagentcli-design.md:560`-`:574`).
- Actual Implementation: Dream sub-agent contexts inherit the normal interactive `PermissionController`. Under default config, `memory_write` is level 1, only level 0 is auto-allowed, and `default_mode` is `ask`; therefore Dream's own `memory_write` calls produce ASK and the worker awaits `permissions.confirm()` in a background dream run.
- Evidence:
  - Dream context passes the active permissions object into the sub-agent context (`myagent/memory/dream.py:349`-`:353`, `:368`-`:386`, `:388`-`:400`).
  - `memory_write` is level 1 in `TOOL_LEVEL_MAP` (`myagent/permissions/controller.py:31`-`:43`).
  - Default permission config auto-allows only level 0 and uses `default_mode="ask"` (`myagent/config/schema.py:27`-`:43`).
  - Permission checking returns ASK when no allow/deny/default-allow rule matches (`myagent/permissions/controller.py:203`-`:224`), and the sub-agent worker awaits `permissions.confirm()` for ASK (`myagent/subagent/worker.py:324`-`:349`).
  - Focused probe result for a default Dream context: `context_has_permissions True`, `memory_write_level1_check ASK`, `default_mode ask`, `auto_allow_levels [0]`.
- Impact: Automatic Dream can prompt the user from a background task in interactive sessions, violating the no-confirmation/silent requirement. In non-TTY sessions, the confirmation path denies by default, so Dream may fail to perform the memory consolidation it was designed to do.
- Suggested Fix: Give Dream a dedicated permission policy that silently allows only its intended memory maintenance operations, for example `memory_write` constrained to the memory store roots, while still denying project-code writes and network/MCP side effects. Add tests that Dream `memory_write` does not call interactive `confirm()` under default config and still routes through `MemoryStore.write()`.
