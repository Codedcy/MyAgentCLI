## Gap 1: [critical] Built-in skill SKILL.md files are missing
- **Category**: missing
- **Section**: §七 技能系统 — 内置技能（计划）
- **Files**: N/A (directory `myagent/skills/builtin/` does not exist)
- **Description**: The design spec lists 6 built-in skills (brainstorming, code-review, systematic-debugging, tdd, writing-plans, executing-plans) with their purposes and resources. The SkillRegistry references a `builtin_dir` at `myagent/skills/builtin/`, but this directory does not exist in the codebase. No built-in SKILL.md files are shipped with the package. As a result, the L2 context contains zero built-in skills, and the skills index shown to the model is empty unless the user installs skills at ~/.myagent/skills/ or .myagent/skills/.
- **Expected**: Six directories under `myagent/skills/builtin/`, each containing a SKILL.md with proper frontmatter and instructions. At minimum: `brainstorming/SKILL.md`, `code-review/SKILL.md`, `systematic-debugging/SKILL.md`, `tdd/SKILL.md`, `writing-plans/SKILL.md`, `executing-plans/SKILL.md`.
- **Actual**: The directory `myagent/skills/builtin/` does not exist. SkillRegistry.scan silently skips the tier.

## Gap 2: [high] Incomplete runtime permission rule adjustments via natural language
- **Category**: incomplete
- **Section**: §五 权限/沙箱系统 — 对话内调整
- **Files**: myagent/permissions/controller.py, myagent/tools/builtin/config_tools.py, myagent/agent/engine.py
- **Description**: The design spec §五 explicitly states that users can adjust permissions mid-conversation in natural language (e.g. "git 命令不再问我了" → auto_allow "git *", "所有操作都不需要确认了" → allow_all, "除了 rm -rf 之外都放行" → allow_all + auto_deny). While PermissionController has `apply_runtime_rule()` that parses natural language rules into allow/deny lists, this method is never invoked by the engine or by any tool. The config_set tool only supports changing `permissions.default_mode` (ask/allow_all toggle). There is no mechanism to add specific commands or paths to auto_allow/auto_deny lists at runtime.
- **Expected**: The Agent engine should route natural language permission change requests to `PermissionController.apply_runtime_rule()`, either by detecting intent patterns in the engine or via a dedicated tool (e.g. expanding config_set or adding permission_allow/permission_deny tools).
- **Actual**: `apply_runtime_rule()` exists but is a dead code path. The only operational mid-conversation permission change is the default_mode toggle via config_set.

## Gap 3: [medium] Session listing output format deviates from design spec
- **Category**: deviation
- **Section**: §十 会话系统与项目感知 — 会话列表展示
- **Files**: myagent/cli/main.py (_print_sessions_rich function)
- **Description**: The design spec shows a specific compact line-based session listing format: one line per session with status icons (✅/📋/—) inline, displaying session ID, first message snippet, duration, token count, and goal status. The implementation renders a Rich table with separate columns for each field. While this is functionally equivalent, the visual layout differs from the specification.
- **Expected**: Output matching the spec format: `2026-07-02-abc123  ✅ "设计 MyAgentCLI 架构"    2.3h  238K tk  Goal ✓`
- **Actual**: Rich table with Session ID, Status, First Message, Duration, Tokens, Goal columns.

## Gap 4: [medium] `/exit --force` and `/quit --force` exit_requested flag is ignored by REPL
- **Category**: incomplete
- **Section**: §十 会话系统与项目感知 — 会话结束流程
- **Files**: myagent/cli/repl.py, myagent/cli/commands.py
- **Description**: The CommandDispatcher._cmd_exit returns `CommandResult(exit_requested=True)` when called with `--force` or `-f`. However, the REPL's `process_input` never checks `result.exit_requested`. It only checks for the literal strings `/exit` and `/quit` to set `self._running = False`. Forced exit with `--force` flag falls through to the dispatcher, receives a success result, and continues running. The `/exit` and `/quit` commands without `--force` are caught by the REPL's direct string check, but the `--force` variant is not handled.
- **Expected**: Either the REPL checks `result.exit_requested` after dispatch, or the dispatcher's exit command is updated to work consistently.
- **Actual**: `result.exit_requested` is set but never consumed by the caller.

## Gap 5: [medium] [[wiki links]] in memory files are detected but never resolved
- **Category**: incomplete
- **Section**: §六 记忆系统 + 梦境机制 — 记忆文件格式
- **Files**: myagent/memory/store.py
- **Description**: The design spec states "正文中可链接到其他记忆文件 [[other-memory-name]]". The MemoryStore.write() method extracts [[wiki-style links]] from memory file bodies using the LINK_RE regex and stores them in the MemoryFile's metadata under the "links" key. However, these links are never resolved into actual cross-references. The recall system doesn't traverse links to find related memories, and the dream engine doesn't use them for contradiction detection or consolidation. The links are effectively parsed and discarded.
- **Expected**: Cross-reference resolution: when recalling memories, follow wiki links to include related memories. Or at minimum, the memory read path should surface the links metadata to the caller.
- **Actual**: Links are parsed and stored in metadata["links"] but never consumed.

## Gap 6: [low] Tool result summarization omits file reference to persisted full result
- **Category**: incomplete
- **Section**: §四 工具系统 — 工具调用流程
- **Files**: myagent/agent/engine.py (_summarize_via_subagent)
- **Description**: The design spec's tool call flow shows: "result > 5000 chars → sub-agent summarizes → returns summary + file reference". The engine's `_summarize_via_subagent` returns `"[Summarized from {len} chars]\n{summary}"` but does not include a reference to the file where the full result is persisted (via `save_tool_call` in the session store's `tools/call-NNN.json`). While the full result IS persisted, the model has no way to know where to find it.
- **Expected**: Summary output includes a reference to the persisted file path, e.g. `"[Summarized from {len} chars. Full result: tools/call-NNN.json]\n{summary}"`
- **Actual**: Summary only includes `"[Summarized from {len} chars]\n{summary}"` with no file reference.

## Gap 7: [low] Prompt log filename format deviates from design spec
- **Category**: deviation
- **Section**: §十一 日志系统 — LLM 交互记录
- **Files**: myagent/llm/provider.py (_write_prompt_logs, _write_response_log)
- **Description**: The design spec specifies prompt log file naming as `2026-07-03T14-32-01-abc123-request.json` and `2026-07-03T14-32-04-abc123-response.json` (ISO 8601 with T separator between date and time). The implementation uses `20260703-143201-nosession-request-0001.json` (compact YYYYMMDD-HHMMSS without T, plus a numeric counter). While functionally equivalent for uniqueness, the format differs from the specification.
- **Expected**: Timestamp format like `2026-07-03T14-32-01-{session_part}-request.json`
- **Actual**: Timestamp format like `20260703-143201-{session_part}-request-{counter:04d}.json`

## Gap 8: [low] Ripgrep --max-count (-m) flag limits matches per-file, not total output
- **Category**: deviation
- **Section**: §四 工具系统 (grep tool specification)
- **Files**: myagent/tools/builtin/search_tools.py
- **Description**: The grep tool's `head_limit` parameter is described as "Limit output to first N lines/entries". In the ripgrep path, this is implemented via the `-m` flag which limits matches per file, not globally. If searching 10 files with `head_limit=5`, ripgrep will return up to 50 matches (5 per file) rather than 5 total. In contrast, the Python fallback applies head_limit to the combined output across all files, capping total entries at the limit. This creates inconsistent behavior between the two search backends.
- **Expected**: Both backends should apply head_limit to the total combined output. The ripgrep path should post-process results with offset+head_limit (as it already does for offset).
- **Actual**: Ripgrep path uses `-m` (per-file limit); Python path uses global head_limit.

## Gap 9: [low] Project detection uses `python` not `python3` for version detection
- **Category**: incomplete
- **Section**: §十 会话系统与项目感知 — 环境感知
- **Files**: myagent/agent/project.py (_detect_python_version)
- **Description**: The ProjectDetector._detect_python_version() runs `python --version` to detect the Python version. On systems where `python` is Python 2.x and `python3` is Python 3.x (e.g. older macOS, some Linux distros), this will return the wrong version or a Python 2 version number. The design spec lists `python_version: "3.12"` as an example, implying Python 3 detection.
- **Expected**: Try `python3 --version` first, then fall back to `python --version`, or use `sys.version` which is always the running interpreter's version.
- **Actual**: Only `python --version` is attempted.
