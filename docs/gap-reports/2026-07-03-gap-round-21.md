## Blocking / Important Gaps

### G21-01
- **Severity**: High
- **Design Requirement**: 权限系统只有配置/运行时规则和 CLI 级 `--dangerously-skip-permissions` 能改变确认策略；普通工具调用应按 auto_deny -> auto_allow -> default_mode 的流程检查权限，权限确认不设超时并等待明确响应。设计证据：`docs/superpowers/specs/2026-07-02-myagentcli-design.md:452`, `docs/superpowers/specs/2026-07-02-myagentcli-design.md:468`, `docs/superpowers/specs/2026-07-02-myagentcli-design.md:474`。
- **Actual Implementation**: `bash` 工具 schema 暴露了模型可填写的 `dangerouslyDisableSandbox` 参数，并且 AgentEngine 在任何工具参数包含该值时跳过集中权限检查。
- **Evidence**: `myagent/tools/builtin/exec_tools.py:35` 暴露 `"dangerouslyDisableSandbox"`；`myagent/agent/engine.py:828` 说明该参数会 bypass；`myagent/agent/engine.py:832` 读取 `tc.params.get("dangerouslyDisableSandbox", False)`；`myagent/agent/engine.py:836` 在 `not skip_perm_check` 时才执行权限检查。
- **Impact**: 模型或 prompt injection 可以在单个工具调用中自授权限，绕过用户确认、auto_deny 和默认 ask 策略，破坏权限系统最核心的安全边界。
- **Suggested Fix**: 从 `BashTool.parameters` 移除该字段；删除 `_execute_tool()` 中由工具参数驱动的 skip 分支。只允许 `PermissionController.skip_all(True)` 这类由 CLI flag 或明确运行时规则控制的全局状态绕过权限，并补测试覆盖“模型传入 dangerouslyDisableSandbox 仍需确认/拒绝”。

### G21-02
- **Severity**: Medium
- **Design Requirement**: MCP 启动读取项目级 + 用户级 `mcp.json`，并应服从项目级配置高于用户级配置的总体配置优先级。设计证据：`docs/superpowers/specs/2026-07-02-myagentcli-design.md:424`, `docs/superpowers/specs/2026-07-02-myagentcli-design.md:793`。
- **Actual Implementation**: CLI 入口解析了 `--project-dir` 并将其用于 config、memory、skills、sessions，但 MCP 启动函数没有接收 `project_dir`，内部使用 `Path.cwd()` 查找项目级 `.myagent/mcp.json`。同时，代码注释说项目级覆盖用户级，但实际先扫描用户级，再用 `seen_servers` 跳过项目级同名 server。
- **Evidence**: `myagent/cli/main.py:55` 设置 `project_dir`；`myagent/cli/main.py:82` 调用 `_startup_mcp_servers(tool_registry)` 未传项目目录；`myagent/cli/main.py:410` 注释声明 project overrides user；`myagent/cli/main.py:411` 先列用户级路径；`myagent/cli/main.py:413` 使用 `Path.cwd()`；`myagent/cli/main.py:429` 和 `myagent/cli/main.py:431` 导致先见到的用户级同名 server 阻止项目级覆盖。
- **Impact**: 使用 `myagent --project-dir X` 时不会加载 `X/.myagent/mcp.json`，而是加载当前 shell 目录下的 MCP 配置；即使在项目目录运行，用户级同名 MCP server 也会压过项目级配置，和项目隔离/项目优先级不一致。
- **Suggested Fix**: 将 `_startup_mcp_servers(tool_registry, project_dir)` 显式传入项目目录；项目路径使用 `project_dir / ".myagent" / "mcp.json"`。合并策略改成低优先级先读、后读项目级覆盖同名 server，或先构建 dict 后按优先级覆盖；增加 CLI `--project-dir` MCP 配置测试和同名 server 覆盖测试。

### G21-03
- **Severity**: Medium
- **Design Requirement**: dream 机制触发后应扫描“未处理的 session transcripts”，分析新约定、反复纠正、过时记忆和矛盾记忆，然后更新 `last_dream.json`。设计证据：`docs/superpowers/specs/2026-07-02-myagentcli-design.md:560`, `docs/superpowers/specs/2026-07-02-myagentcli-design.md:574`。
- **Actual Implementation**: dream 触发条件已按 `last_run` 过滤轮数，但真正扫描 transcript 时仍只按最近 7 天和固定 top N 取样：sub-agent prompt 只取最近 3 个，inline 扫描只取最近 5 个；没有使用 `last_run`、processed transcript ID 或 last processed timestamp 来界定“未处理”。
- **Evidence**: `_build_transcript_excerpts()` 使用 7 天 cutoff 和 `recent_sessions[:3]`，见 `myagent/memory/dream.py:269`, `myagent/memory/dream.py:285`；`_scan_transcripts()` 使用 7 天 cutoff 和 `recent_sessions[:5]`，见 `myagent/memory/dream.py:702`, `myagent/memory/dream.py:723`；dream state 只写 `last_run`/`round_count`，见 `myagent/memory/dream.py:218`, `myagent/memory/dream.py:219` 和 `myagent/memory/dream.py:678`。
- **Impact**: dream 可能重复扫描已经处理过但仍在最近 7 天内的 sessions，也可能漏掉超过 top N 或超过 7 天但从未处理过的 transcripts。记忆整合结果会不稳定，尤其是长时间运行或大量会话后。
- **Suggested Fix**: 在 `last_dream.json` 中记录 `last_processed_at` 或 processed transcript IDs；扫描条件基于 transcript `created_at`/`updated_at` 与 last processed marker，而不是固定 7 天/top N。保留 top N 只能作为 prompt 截断策略，不能作为“未处理 transcript”的发现边界；补测试覆盖已处理/未处理 session 集合。

### G21-04
- **Severity**: Medium
- **Design Requirement**: `send_message` 用于主 Agent 与子 Agent 双向通信；子 Agent 发送到主 Agent 时，主 Agent 需要知道消息来自哪个子 Agent，才能在并发子任务下正确归因和继续编排。设计证据：`docs/superpowers/specs/2026-07-02-myagentcli-design.md:354`, `docs/superpowers/specs/2026-07-02-myagentcli-design.md:761`。
- **Actual Implementation**: `send_message` schema 声称 `from` 会在子 Agent 上下文中自动填充，但实际执行缺省时仍写死为 `"subagent"`；`ToolContext` 没有当前子 Agent ID 字段，worker 也未在调用工具前注入 sender ID。
- **Evidence**: schema 描述“populated automatically”，见 `myagent/tools/builtin/agent_tools.py:84`；执行时 `params.get("from", "subagent")`，见 `myagent/tools/builtin/agent_tools.py:115`；`ToolContext` 字段没有 current subagent id，见 `myagent/tools/base.py:54`；worker 复用 parent tool context/仅传 subagent_pool，见 `myagent/subagent/worker.py:256` 和 `myagent/subagent/worker.py:261`。
- **Impact**: 多个子 Agent 同时向主 Agent 发送消息时，主 Agent drain 到的消息可能都显示来自 `"subagent"`，无法可靠区分来源，影响中途介入、汇总和状态展示。
- **Suggested Fix**: 给 `ToolContext` 增加 `current_subagent_id`，由 `SubAgentPool._run_background/_run_foreground` 创建 worker 时填入；或在 `SubAgentWorker` 执行 `send_message(to="main")` 前注入 `params["from"] = handle.id`。补一个子 Agent 省略 `from` 仍能入队真实 ID 的测试。

### G21-05
- **Severity**: Medium
- **Design Requirement**: 子 Agent 池最大并发为 `min(16, os.cpu_count() - 2)`，全局上限为单会话 1000。设计证据：`docs/superpowers/specs/2026-07-02-myagentcli-design.md:727`, `AGENTS.md:59`。
- **Actual Implementation**: `SubAgentPool` 在 `max_concurrent is None` 时会按 CPU 公式计算，但 CLI 主路径总是传入 `config.subagents.max_concurrent`；默认 schema 值是 10，因此自动 CPU 公式在默认运行路径不会生效。
- **Evidence**: 默认配置为 `10`，见 `myagent/config/schema.py:49`；CLI 总是传该值，见 `myagent/cli/main.py:117` 和 `myagent/cli/main.py:118`；CPU 公式只在 `max_concurrent is None` 分支执行，见 `myagent/subagent/pool.py:169` 和 `myagent/subagent/pool.py:170`。
- **Impact**: 在小 CPU 机器上可能超过 `cpu_count - 2`，在大 CPU 机器上又低于设计默认上限 16。实际调度行为与设计的资源保护/并发能力不一致。
- **Suggested Fix**: 将 `SubagentsConfig.max_concurrent` 设计成 `None | int`，默认 `None` 表示 auto；或者在配置加载后将用户值 clamp 到 `min(16, os.cpu_count() - 2)`。同步更新 schema、配置测试和池初始化测试。

### G21-06
- **Severity**: Medium
- **Design Requirement**: 日志系统要求任何异常使用 `category="error"` 并包含 `exception_type`、`traceback`、`context` 和 `component`；AGENTS.md 也要求所有 except 块记录这些字段。设计证据：`docs/superpowers/specs/2026-07-02-myagentcli-design.md:984`, `AGENTS.md:103`。formatter 只有在 `record.exc_info` 存在时才写入 `exception_type`/`traceback`，见 `myagent/logging/formatter.py:80`。
- **Actual Implementation**: 多处 except 块直接 `return ToolResult(error=str(e))`、`continue` 或 `pass`，没有 `logger.exception`/`exc_info=True`，因此不会产生带 traceback 的 `error` category 日志。
- **Evidence**: `BashTool.execute()` 捕获通用异常直接返回，见 `myagent/tools/builtin/exec_tools.py:86`；`ReadTool/WriteTool/EditTool/GlobTool` 捕获异常直接返回，见 `myagent/tools/builtin/file_tools.py:84`, `myagent/tools/builtin/file_tools.py:131`, `myagent/tools/builtin/file_tools.py:201`, `myagent/tools/builtin/file_tools.py:236`；`SpawnSubagentTool` 和 `SendMessageTool` 捕获异常直接返回，见 `myagent/tools/builtin/agent_tools.py:70`, `myagent/tools/builtin/agent_tools.py:125`；`DreamEngine` 也存在静默 continue/pass 的扫描路径，见 `myagent/memory/dream.py:234`, `myagent/memory/dream.py:300`, `myagent/memory/dream.py:1539`。
- **Impact**: 用户仍可能看到工具错误，但结构化日志缺少异常类型、traceback 和组件上下文，无法满足可审计性和故障诊断要求；线上问题会很难从 JSONL 日志复盘。
- **Suggested Fix**: 为所有非预期异常路径使用 `logger.exception(..., extra={"category": "error", "component": "...", "context": "..."})` 或 `logger.error(..., exc_info=True, extra=...)`。对用户可预期错误（文件不存在、regex 无效等）可以继续返回 `ToolResult(error=...)`，但捕获 `Exception` 的兜底路径必须记录。

## Minor Gaps / Documentation Gaps
- MCP 协议技术选型表写的是 `mcp (Python SDK)`，依赖中也包含 `mcp>=1.0.0`，但当前 `myagent/tools/mcp/client.py` 是手写 JSON-RPC transport/client，未看到 `import mcp`。功能层面已有 stdio/SSE 抽象和工具适配；若继续保留手写客户端，建议在设计文档或 README 中澄清“使用 MCP 协议，不直接依赖 SDK client API”。

- 四层模块结构基本落地：`myagent/cli`、`agent`、`tools`、`subagent`、`context`、`memory`、`skills`、`session/config/permissions/llm/logging` 均存在并有对应测试目录。
- 配置 schema 默认值与设计中的核心项基本一致：DeepSeek、Think High、context compression、permissions、dream、tools、ui、session、logging 等见 `myagent/config/schema.py`。
- ConfigLoader 实现了默认值、用户 AGENT、用户 config、项目 AGENT、项目 config、runtime override、CLI override 的 deep-merge，且列表替换策略有测试覆盖。
- AgentEngine 维持单 ReAct loop，goal 未达成会注入 remaining work 后继续循环；`tests/agent/test_engine.py:166` 覆盖该路径。
- LLMProvider 支持 streaming、thinking mode 映射、fallback models、request/response 日志、prompt debug 文件和显式 retry/backoff。
- ContextBuilder/CompressionEngine 实现 L0-L6 组装、技能索引/active skill 注入、记忆召回缓存、4 层渐进压缩、summary/tool result 持久化。
- ToolRegistry 和内置工具清单基本覆盖设计；MCP 工具 adapter、resource/prompt reference 和 read/get prompt 工具已实现。
- SessionStore 实现 `<project-name>/<project-hash>/<session-id>/`、JSON/Markdown transcript、tools/summaries、resume/list/export 和 closed marker。
- MemoryStore 使用 frontmatter Markdown、MEMORY.md index、去重更新、session write log；recall 有语义/关键词 fallback。
- Skills 实现内置/用户/项目三级发现、项目覆盖、`skill_invoke` 虚拟工具、`/skill-name` 强制调用和资源相对路径。
- CLI 使用 prompt_toolkit/Rich，支持 `/mode`、`/goal`、`/skills`、`/dream`、`/clear`、`/history`、`/exit`，并有 status bar 与子 Agent 状态回调。
