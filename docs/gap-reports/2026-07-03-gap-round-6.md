# Gap Report — Round 6

## Summary
- 缺失功能 (missing): 0
- 不完整实现 (incomplete): 6
- 偏离设计 (deviation): 2

All four architectural layers, all 13 design-spec tools, all six config sections, all log categories, the full ReAct loop with compression, the dream engine, the permission system, and the skill registry are implemented. The gaps below are refinements: a mechanism mismatch in intent detection, a missing model-to-skill bridge, incomplete sub-agent context filtering, and several minor incompleteness issues.

## Gap 1: Intent detection uses keyword matching instead of model-driven interpretation
- **Category**: deviation
- **Severity**: medium
- **Section**: §二 核心Agent循环 — 用户交互
- **Files**: `myagent/agent/engine.py` (lines 587–638, `_detect_intent`)
- **Description**: The design spec states that the model "自行理解意图" (self-judges user intent from natural language) during the Think phase. The spec table maps user utterances like "别改了，停下来" → "停止当前操作" and "不对，你应该先用 pytest" → "方向纠正", implying the model recognizes intent as part of NLU, not post-hoc text matching. The current implementation applies hard-coded keyword lists (`stop_phrases`, `correct_phrases`, `insert_phrases`, `continue_phrases`) to the model's own output text to classify intent signals. This means intent detection is a regex/string-match layer bolted onto the engine rather than delegated to the model's reasoning.
- **Expected**: The model, given appropriate system instructions, recognizes user intent during the Think phase and signals stop/correct/insert/continue through structured output or standard conversational cues that the engine interprets. No keyword list should be hard-coded in the engine.
- **Actual**: `_detect_intent()` pattern-matches on the model's response text using 30+ hard-coded English and Chinese keywords. The model is not instructed about intent signaling; the engine guesses intent from model output strings.

## Gap 2: Sub-agent L3 project context passed all-or-nothing, not filtered for task relevance
- **Category**: incomplete
- **Severity**: medium
- **Section**: §三 上下文分层结构 — 主 Agent vs 子 Agent 上下文
- **Files**: `myagent/subagent/worker.py` (lines 111–121), `myagent/subagent/pool.py` (lines 191–199)
- **Description**: The design spec states that for sub-agents, L3 project context is "按需传递（仅传递与子任务相关的项目上下文）" — only task-relevant project context should be passed. The current implementation passes the full `project_context` object when available at spawn time, injecting all detected fields (project_type, git_branch, structure_summary) into the sub-agent's system prompt without filtering based on the sub-task prompt.
- **Expected**: The spawn flow should extract only the project-context fields relevant to the specific sub-task (e.g., for a code-review sub-task, pass git_branch and project_type but not necessarily test_framework if irrelevant). This could be done via a lightweight relevance check or by passing the project context as reference data the sub-agent can query.
- **Actual**: If `project_context` is provided in the `ToolContext` at spawn time, all fields are unconditionally injected into the sub-agent's system prompt via `_run_impl()`.

## Gap 3: skill_invoke virtual tool not communicated to the model — natural language skill invocation cannot work
- **Category**: incomplete
- **Severity**: medium
- **Section**: §七 技能系统 — 技能调用实现机制
- **Files**: `myagent/agent/engine.py` (lines 379–417), `myagent/context/builder.py` (L0_SYSTEM_PROMPT, lines 64–78)
- **Description**: The design spec describes a `skill_invoke` virtual tool: the model outputs `tool_call(name="skill_invoke", params={"skill": "<name>"})`, and the engine intercepts it to load the skill's full SKILL.md content. The spec explicitly says `skill_invoke` does NOT appear in L1 tools schema. For the model to emit this tool call, it must know about `skill_invoke` from somewhere. Currently, neither the L0 system prompt nor the L2 skills index mentions `skill_invoke` as a mechanism. The only working path is forced invocation via `/skill-name` in the REPL (handled by `CommandDispatcher`). Natural-language auto-invocation — where the model sees a matching skill in L2 and decides to invoke it — cannot work because the model has no way to express the invocation.
- **Expected**: Either (a) include `skill_invoke` in the L1 tools schema (which the spec says not to do), or (b) document in the system prompt that the model should output `tool_call(name="skill_invoke", ...)` when it determines a skill is relevant. The latter preserves the spec's intent while giving the model the information it needs.
- **Actual**: `skill_invoke` handling exists in `AgentEngine._react_loop()` (lines 379–417) but the model receives no instruction about its existence. The L0 system prompt mentions skills only indirectly; L2 lists skill names and descriptions. The engine correctly intercepts `skill_invoke` tool calls, but the model will never produce them.

## Gap 4: Transcript message content truncated at 5000 chars per message
- **Category**: incomplete
- **Severity**: low
- **Section**: §三 上下文分层结构 — 持久化策略
- **Files**: `myagent/context/persistence.py` (line 374, `_write_transcripts`)
- **Description**: The spec requires "所有记录必须持久化，确保可追溯" (all records must be persisted for traceability). In `SessionStore._write_transcripts()`, individual message content is truncated to 5000 characters (`m.content[:5000]`). While full tool call results are persisted separately in `tools/call-NNN.json`, long text responses from the model (e.g., generated code, long explanations) are silently truncated in the transcript. This breaks full traceability.
- **Expected**: Either persist full message content without truncation, or store truncated content in the main transcript with a reference to a separate full-content file (similar to how tool calls are handled).
- **Actual**: Both JSON and Markdown transcripts have `content[:5000]` for each message.

## Gap 5: send_message tool has no entry in TOOL_LEVEL_MAP
- **Category**: incomplete
- **Severity**: low
- **Section**: §五 权限/沙箱系统 — 权限分级
- **Files**: `myagent/permissions/controller.py` (lines 32–47, `TOOL_LEVEL_MAP`)
- **Description**: The `send_message` tool is one of the 13 built-in tools listed in the design spec, but it has no entry in `TOOL_LEVEL_MAP`. This causes it to default to permission level 3 (network_write — the most restrictive fallback level). The spec's level table doesn't explicitly classify `send_message`, but given its read-only nature (sending messages within the agent system, no filesystem/network side effects), it should reasonably be classified at level 0 or level 2 alongside `spawn_subagent`.
- **Expected**: `send_message` should have an explicit entry in TOOL_LEVEL_MAP at an appropriate level (level 0 as a read-only communication tool, or level 2 alongside spawn_subagent).
- **Actual**: `send_message` is missing from TOOL_LEVEL_MAP; falls through to `_get_level()` returning the default of 3.

## Gap 6: Memory cache never expires within a session — stale on topic change
- **Category**: incomplete
- **Severity**: low
- **Section**: §六 记忆系统 + 梦境机制 — 记忆生命周期
- **Files**: `myagent/context/builder.py` (lines 86–87, `_memory_cache` and `_cache_key`)
- **Description**: The spec says memories are loaded at startup via "语义匹配当前任务 → 加载相关记忆到 L4". `ContextBuilder.build()` computes a `_cache_key` from the first 100 characters of the initial user input and caches the recall results permanently for the session. If the user changes topics mid-session (e.g., starts discussing code style, then switches to deployment configuration), the cache hit returns the initial topic's memories instead of re-running semantic recall against the new topic.
- **Expected**: Either expire the cache based on topic drift detection, re-run recall periodically, or use a sliding window of recent queries to determine the cache key.
- **Actual**: `_memory_cache` uses a single key derived from the first input and never refreshes.

## Gap 7: Dream engine round counting misses in-memory current session during mid-session trigger check
- **Category**: incomplete
- **Severity**: low
- **Section**: §六 记忆系统 + 梦境机制 — 触发条件
- **Files**: `myagent/agent/session.py` (lines 53–84, `estimate_total_rounds`), `myagent/cli/repl.py` (lines 441–478, `_periodic_dream_check`)
- **Description**: The dream trigger requires cumulative conversation rounds > 50 (AND distance from last dream > 6 hours). `SessionManager.estimate_total_rounds()` reads `turn_count` from saved `transcript.json` files across all sessions. However, during a long-running session, the periodic dream checker (`_periodic_dream_check`, every 30 minutes) calls `estimate_total_rounds()` which reads only persisted transcripts — the current session's in-memory (unsaved) rounds are not counted. This means the dream could fail to trigger when it should (e.g., the current session generates 60 rounds, but the last saved transcript only shows 40).
- **Expected**: The round estimation should include the current session's live turn count when available, e.g., `session.turn_count` from the active session object.
- **Actual**: Only persisted transcript files are counted; the active session's rounds are invisible to the checker until the session ends and the transcript is written.

## Gap 8: Sub-agent status bar updates only on spawn, not on completion or status change
- **Category**: incomplete
- **Severity**: low
- **Section**: §八 子 Agent 池与工作流编排 — Context 状态栏展示
- **Files**: `myagent/cli/main.py` (lines 186–248, `_spawn_with_status`), `myagent/cli/status.py`, `myagent/subagent/pool.py`
- **Description**: The design spec shows a status bar with live sub-agent details — running (with progress %), completed (with result summary), retrying (with retry count). The current implementation monkey-patches `SubAgentPool.spawn` to trigger status bar updates, but this wrapper only executes when `spawn()` is called. Sub-agent completions, failures, or status transitions that occur during execution do not trigger status bar refreshes. The status bar will show stale state until the next `spawn()` call.
- **Expected**: Status bar updates should be triggered by sub-agent lifecycle events (completion, failure, interruption), not just spawn. This could be done via callbacks on `SubAgentHandle` or by having the pool notify the status bar on state transitions.
- **Actual**: `_spawn_with_status` wraps `pool.spawn` and calls `status_bar.update()` after each spawn. No hooks exist for completion/status-change notifications.
