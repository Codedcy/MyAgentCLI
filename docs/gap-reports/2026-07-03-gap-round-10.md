## Gap 1: [medium] AgentEngine.TOOL_RESULT_MAX_CHARS hardcoded — ignores config.tools.tool_result_max_chars
- **Category**: deviation
- **Section**: §四 工具系统, §九 配置系统
- **Files**: `myagent/agent/engine.py`
- **Description**: `AgentEngine` uses a hardcoded class constant `TOOL_RESULT_MAX_CHARS = 5000` for the large-result summarization threshold (line 95) and for fallback truncation in `_truncate_result` (line 906). The config schema defines `tools.tool_result_max_chars` (default 5000), and the `CompressionEngine` correctly reads this config value for Layer 2 compression, but the `AgentEngine` never reads it. If a user changes `tool_result_max_chars` to e.g. 10000 in config.yaml, the engine still truncates at 5000 chars.
- **Expected**: AgentEngine should read `self.config.tools.tool_result_max_chars` (or accept it as a constructor parameter) and use it as the threshold in both `_execute_tool` and `_truncate_result`.
- **Actual**: Hardcoded class constant `TOOL_RESULT_MAX_CHARS = 5000` on line 95, used unconditionally at lines 829 and 906.

## Gap 2: [medium] Sub-agent handles accumulate indefinitely in pool — no periodic cleanup
- **Category**: deviation
- **Section**: §八 子 Agent 池与工作流编排
- **Files**: `myagent/subagent/pool.py`
- **Description**: `SubAgentPool._agents` dict (line 151) accumulates all spawned sub-agent handles and never removes completed, failed, or interrupted entries during the session. Handles are only removed on `shutdown()` (line 276), which runs at process exit. The status bar callback iterates the entire `_agents` dict on every lifecycle event (line 203 in main.py), meaning completed agents remain in the status bar panel for the entire session. Over long sessions with hundreds of sub-agents, this wastes memory and presents stale information to the user.
- **Expected**: Periodically remove handles that have reached terminal states (COMPLETED, FAILED, INTERRUPTED) after a configurable retention period, or remove them from the dict immediately after the status bar callback has consumed the final state.
- **Actual**: `_agents` dict grows monotonically until process exit. No entries are ever removed from it except via `shutdown()`.

## Gap 3: [medium] Sub-agent model override parameter accepted but ignored
- **Category**: incomplete
- **Section**: §八 子 Agent 池与工作流编排 (spawn_subagent tool, model parameter)
- **Files**: `myagent/subagent/worker.py`
- **Description**: `SubAgentWorker.__init__` accepts a `model` parameter (line 82: `model: str | None = None`) and stores it as `self.model`. However, in `_run_impl` (line 118), the worker calls `self.llm.complete(...)` without ever referencing `self.model` to switch the underlying LLM provider's model. The `spawn_subagent` tool and `SubAgentPool.spawn()` both accept and forward a `model` parameter, but it has no runtime effect — the sub-agent always uses the same LLM provider model as the main agent.
- **Expected**: When `model` is provided, the Worker should either configure a new LLMProvider instance for that model, or pass a `model` parameter to `self.llm.complete()` that the LLMProvider uses to override its default model for that call.
- **Actual**: `self.model` is assigned (line 82) but never read in the execution path.

## Gap 4: [low] MCP server stderr output logged at DEBUG only — invisible at default INFO level
- **Category**: incomplete
- **Section**: §四 工具系统 — MCP 集成
- **Files**: `myagent/tools/mcp/client.py`
- **Description**: `StdioTransport._drain_stderr` (line 110) reads MCP server stderr line-by-line and logs every line at `logger.debug()` level (line 119). At the default `INFO` log level, all MCP server stderr output is invisible. If an MCP server crashes, emits warnings, or logs diagnostic information to stderr (not JSON-RPC error responses), the user sees no indication of the problem.
- **Expected**: MCP stderr output should be logged at a configurable level. Warnings/errors on stderr (lines containing "error", "warning", "traceback") should be logged at WARNING or ERROR level so they are visible at default INFO settings.
- **Actual**: All stderr output is `logger.debug()` — completely invisible at INFO level.

## Gap 5: [low] `--export` CLI argument lacks format validation
- **Category**: incomplete
- **Section**: §十 会话系统与项目感知
- **Files**: `myagent/cli/main.py`
- **Description**: The `--export` CLI argument (line 22) accepts any string value without validating it against supported formats. Passing an unrecognized format like `--export pdf` silently falls through `export_session()` to produce a JSON export file (since only `"markdown"` has a dedicated code path; all other values are treated as JSON). The user gets a file named `...-export.json` with no error or warning that their requested format was not honored.
- **Expected**: Either add `choices=["markdown", "json"]` to the argparse definition, or validate the format value in `async_main` before calling `export_session()` and print an error for unrecognized formats.
- **Actual**: No format validation; `parser.add_argument("--export", ...)` has no `choices` constraint.

## Gap 6: [low] Tool call persistence truncates to 50000 chars — not fully "完整" per spec
- **Category**: deviation
- **Section**: §三 上下文分层结构 (持久化策略: tools/call-NNN.json — "单个工具调用完整输入/输出"), §四 工具系统
- **Files**: `myagent/context/persistence.py`
- **Description**: `SessionStore.save_tool_call` (line 110) truncates the persisted tool call result to 50000 characters: `"result": str(call.result)[:50000]`. The design spec states that `tools/call-NNN.json` stores "完整输入/输出" (complete input and output). While 50000 chars is generous, it is technically not "完整" (complete) for very large tool results (e.g., reading a large file or a comprehensive grep output).
- **Expected**: Store the complete result without truncation, or use the long-messages reference file pattern already implemented in `_persist_message_content` for transcript messages (lines 380-412 in persistence.py).
- **Actual**: Hard truncation at `[:50000]`.

## Gap 7: [low] Sub-agent Worker fallback ToolContext has no config — sub-agents without tool_context ignore shell_timeout_seconds
- **Category**: incomplete
- **Section**: §八 子 Agent 池与工作流编排, §四 (bash tool timeout)
- **Files**: `myagent/subagent/worker.py`
- **Description**: In `SubAgentWorker._run_impl` (line 234), when `self.tool_context` is None, a fallback `ToolContext` is created with `config=None`. When the BashTool executes in a sub-agent via this fallback path, it reads `context.config.tools.shell_timeout_seconds` (exec_tools.py line 48), which is `None` because config is None. The BashTool falls back to a hardcoded 120000ms default. This means `config.tools.shell_timeout_seconds` has no effect on sub-agents spawned without explicit tool_context.
- **Expected**: The Worker should receive and pass the app config, either through the tool_context parameter or via an additional config parameter, so that sub-agents respect the configured shell timeout.
- **Actual**: The fallback `ToolContext` is constructed with `config=None` on line 237.
