## Gap 1: [medium] web_fetch uses full primary model instead of "small fast model" for prompt-guided Q&A

- **Category**: deviation
- **Section**: §四 工具系统 — web_fetch
- **Files**: `myagent/tools/builtin/web_tools.py`
- **Description**: The design spec states that web_fetch should "answer `prompt` against it using a small fast model". The implementation (`_llm_answer` method, line 196-276) uses `litellm.acompletion` with the configured primary model (`deepseek-v4-pro`) in Non-think mode. It does not use a distinct lightweight/fast model for content Q&A.
- **Expected**: web_fetch should use a distinct small, fast model (not the primary DeepSeek V4 Pro) for prompt-guided extraction from fetched web content. The full model call should be reserved for tasks that need deep reasoning.
- **Actual**: The code retrieves the model from `context.config.model` and uses the exact same model as all other LLM calls. The `_WEB_FETCH_ANSWER_MODEL_MODE = "Non-think"` and `_WEB_FETCH_MAX_CONTENT_CHARS = 15000` constants help with performance but do not change the model size/cost.

## Gap 2: [low] No `/export` slash command in the REPL

- **Category**: missing
- **Section**: §十 会话系统与项目感知, §一 CLI Layer (Slash Commands)
- **Files**: `myagent/cli/commands.py`
- **Description**: The design spec lists session export as a capability via `myagent --session <session-id> --export markdown`. Users can export only via the CLI startup flag. There is no `/export markdown` or `/export json` slash command available from within the running REPL session.
- **Expected**: A `/export [markdown|json]` slash command handler in `CommandDispatcher` that exports the current session transcript to the export/ subdirectory, matching the CLI flag behavior but usable mid-session.
- **Actual**: `commands.py` has no export handler. The `--export` flag is handled only in `main.py` at startup (line 178-181), which requires knowing the session ID ahead of time and cannot be used interactively.

## Gap 3: [medium] Sub-agent retry progress not wired from worker to status bar

- **Category**: incomplete
- **Section**: §八 子 Agent池, §一 Status Bar
- **Files**: `myagent/subagent/worker.py`, `myagent/subagent/pool.py`, `myagent/cli/status.py`
- **Description**: The `SubAgentInfo` dataclass has `retry_count` and `max_retries` fields, and `StatusBar._format_subagent_detail` handles the `"retrying"` status with a display format like `"🔄 retrying (1/3)"`. However, `SubAgentWorker._stream_llm_with_retry` never calls the progress callback or any pool notification to report retry progress. The worker's `progress_callback` (line 166-170) only reports iteration progress (`(cur_iter, max_iter)`), not retry counts.
- **Expected**: During LLM retries, the worker should report retry progress to the pool, which should update the SubAgentInfo with `status="retrying"`, `retry_count`, and `max_retries`, so the status bar renders the retry state as shown in the spec.
- **Actual**: SubAgentInfo retry fields are never populated. The `_stream_llm_with_retry` method handles retries internally with logging but does not notify the pool/status bar. The status bar will never show sub-agents in a "retrying" state.

## Gap 4: [low] LogManager.setup startup event emitted without session_id

- **Category**: incomplete
- **Section**: §十一 日志系统
- **Files**: `myagent/cli/main.py`, `myagent/logging/logger.py`
- **Description**: The spec requires every log entry to carry `session_id`. `LogManager.setup()` emits a "Logging initialized" startup event (line 238-247 in logger.py) with `config_hash`, `python_version`, and `platform` metadata. However, `main.py` calls `LogManager.setup(config=config.logging)` without passing a `session_id`. The session_id is set later via `set_context()` after session creation in the REPL. The startup event in the log file will have `session_id: null`.
- **Expected**: The startup log event should include `session_id` when a session is being created or resumed. The `session_id` should be passed to `LogManager.setup()` or the setup should be deferred until after session creation.
- **Actual**: `LogManager.setup()` is called at line 63 of `main.py`, before the session is created at line 115-117 of `repl.py`. The startup event always has `session_id: null`.

## Gap 6: [medium] Layer 2 compression uses raw truncation instead of semantic summarization

- **Category**: deviation
- **Section**: §三 上下文分层结构 — 压缩策略
- **Files**: `myagent/context/compression.py`
- **Description**: The design spec defines Layer 2 compression as "摘要化 (低成本): 工具结果 → 语义摘要 + 引用". This means tool results should be semantically summarized (using the LLM), not merely character-truncated. The implementation in `_layer2_summarize` (lines 194-221) simply truncates tool result content at `max_chars` with a `[Summarized: N chars → M chars]` prefix. No semantic understanding is applied.
- **Expected**: Layer 2 should use the LLM (in a lightweight mode, similar to the spec's description of "低成本") to produce a semantic summary of each large tool result, preserving key information while discarding redundancy. The summary should include a file reference to the persisted full result.
- **Actual**: Raw character truncation at `tool_result_max_chars` (default 5000). The spec's "语义摘要 + 引用" pattern is not implemented in Layer 2. Note: the engine's `_summarize_via_subagent` method does perform LLM summarization of tool results during execution (at the execution stage), but the distinct compression Layer 2 path does not.

## Gap 7: [medium] Session resume does not restore goal to GoalTracker

- **Category**: incomplete
- **Section**: §十 会话系统, §二 核心Agent循环
- **Files**: `myagent/cli/main.py`, `myagent/agent/goal.py`
- **Description**: When resuming a session with `--resume`, the `Session` object is loaded from `transcript.json` and contains the `goal` field. However, `goal_tracker.set_goal()` is never called with the session's stored goal. The `GoalTracker` was created earlier and only set if `args.goal` was provided as a CLI flag. On resume, the goal tracker remains empty (`get_goal()` returns `None`), so the ReAct loop will not perform goal achievement checks even if the resumed session was a goal-oriented task.
- **Expected**: After loading a resumed session, the code should call `goal_tracker.set_goal(session.goal)` if `session.goal` is not None.
- **Actual**: `main.py` line 151: `if args.goal: goal_tracker.set_goal(args.goal)`. This only sets the goal from CLI args, never from the restored session object. The `session.goal` field is loaded from disk (persistence.py line 179) but never propagated into the live `GoalTracker`.
